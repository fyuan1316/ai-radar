## 调度 & 资源分配

### Kueue
两条补丁线发布 + v0.18 主线大量推进(窗口内 merge 约 157 个 PR,v0.18 GA 临近)。

补丁线 [v0.17.3](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.17.3) / [v0.16.8](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.16.8)(均 2026-05-15):
- 安全修复:两个 podset 总请求超过 int64 导致整数溢出、击穿配额上限([#11140](https://github.com/kubernetes-sigs/kueue/pull/11140) / [#11151](https://github.com/kubernetes-sigs/kueue/pull/11151))—— 配额隔离的安全洞,建议尽快升补丁线
- [FinishOrphanedWorkloads feature gate 从 Beta 降回 Alpha](https://github.com/kubernetes-sigs/kueue/pull/11018) — 修复 Kueue 误把新建 Workload 标记为 finished、阻塞队列的回归
- 多个 TAS 修复:NodeHotSwap 用有效 tolerations(含 AdmissionCheck 注入的)评估节点 taint([#11228](https://github.com/kubernetes-sigs/kueue/pull/11228))、多资源 workload 二次调度时配额被重复计数([#11039](https://github.com/kubernetes-sigs/kueue/pull/11039))、TAS 缓存清理与快照性能优化
- ElasticJobs:相同创建时间戳的 workload slice 排序错误,扩容时引发配额泄漏([#11201](https://github.com/kubernetes-sigs/kueue/pull/11201))

v0.18 主线新能力(本周 merge,面向 v0.18 GA;v0.18.0-rc.1 已于窗口前 2026-05-06 发出):
- [DRA 集成 → Beta(#10996)](https://github.com/kubernetes-sigs/kueue/pull/10996) — KueueDRAIntegration alpha→beta、v0.18 默认开启,覆盖基于 ResourceClaimTemplate + deviceClassMappings 的 DRA 配额核算;同时把 Kueue 的 DRA feature gate 改名以避免与上游 K8s gate 冲突([#11255](https://github.com/kubernetes-sigs/kueue/pull/11255))
- [MultiKueue Global Quotas(KEP-9988)](https://github.com/kubernetes-sigs/kueue/pull/10145) — KEP 合入,并实现 Alpha1 阶段([#11141](https://github.com/kubernetes-sigs/kueue/pull/11141)),提供 manager 级、跨 worker 集群的全局配额
- [ElasticJobsViaWorkloadSlices 走向 Beta](https://github.com/kubernetes-sigs/kueue/pull/11300) — 补齐 beta 毕业所需的 webhook 校验
- [WorkloadPriorityClass 默认化(KEP-10765)](https://github.com/kubernetes-sigs/kueue/pull/10796) — 给未带 `kueue.x-k8s.io/priority-class` 标签的 workload 自动套用名为 default 的 WorkloadPriorityClass,避免把 pod 调度优先级与排队优先级混为一谈,沿用 LocalQueueDefaulting 的约定式模式
- [TAS 软亲和(#10903)](https://github.com/kubernetes-sigs/kueue/pull/10903) — 支持 `preferredDuringSchedulingIgnoredDuringExecution` 节点亲和(TASPreferredSchedulingAffinity gate,软亲和优先于容量适配);并在均衡放置失败时回退 BestFit([#11136](https://github.com/kubernetes-sigs/kueue/pull/11136))
- [Kueue priority booster 示例控制器(#9959)](https://github.com/kubernetes-sigs/kueue/pull/9959) — 给长时间排队的 workload 自动抬优先级,防饿死
- MultiKueue 韧性加固:挂死的 remote watch 不再拖垮全集群准入([#11207](https://github.com/kubernetes-sigs/kueue/pull/11207)),挂死探测从 10 分钟缩到 1 分钟([#11304](https://github.com/kubernetes-sigs/kueue/pull/11304)),单个慢 remote 不再阻塞其它集群([#11305](https://github.com/kubernetes-sigs/kueue/pull/11305))
- [TrainJob](https://github.com/kubernetes-sigs/kueue/pull/11310) / [RayJob](https://github.com/kubernetes-sigs/kueue/pull/11355) 集成移除全局状态 —— Kubeflow Trainer v2 与 Ray 集成的内部清理,为多实例/并发可靠性铺路

启示:v0.18 是 Kueue 把"DRA + 配额"这条链路打通并默认启用的版本。此前对 GPU 配额要么靠 `nvidia.com/gpu` 扩展资源、要么手动开 DRA alpha gate;v0.18 之后,只要工作负载用 ResourceClaimTemplate 申请设备,Kueue 就能直接核算配额。我们若做 GPU 多租户/配额,产品排期可对齐 v0.18 GA;但升级脚本里的 DRA feature gate 名要按 #11255 同步改。MultiKueue 这周既加全局配额又补韧性,多集群 AI 平台方向值得持续跟。

### DRA(Dynamic Resource Allocation)
> 任务数据源里列的 `kubernetes-sigs/dra-evolution` 现已无法访问(GitHub API 返回 404)。该仓库原为 DRA API 原型试验场,DRA 的设计讨论现已全部回归 `kubernetes/enhancements`。本节据此只看 KEP。

本周 KEP 动向(均在 k/enhancements):
- [新增 KEP-6080:DRA Derived Attributes(#6081,2026-05-15)](https://github.com/kubernetes/enhancements/pull/6081) — 给 DeviceRequest 加 scoped CEL 表达式 `derivedAttributes`,在调度时动态合成"虚拟分组键",实现更灵活的设备 co-allocation
- [新增 KEP-6072:DRA Standard numaNode Device Attribute(#6073,2026-05-12)](https://github.com/kubernetes/enhancements/pull/6073) — 把 `resource.kubernetes.io/numaNode` 标准化为 well-known DRA 属性,与 `pcieRoot`/`pciBusID` 并列。NVIDIA、AMD GPU 已各自上报 numa 但命名不一,统一后可跨驱动用 matchAttribute 做 GPU/NIC 共置
- [KEP-5677:partitionable + consumable devices](https://github.com/kubernetes/enhancements/pull/6068) — 规划在 1.37 再走一轮 Alpha,先修分区+可消耗设备的核算 bug 再谈 Beta
- [KEP-5517:DRA Node Allocatable Resources alpha2 改动(#6082)](https://github.com/kubernetes/enhancements/pull/6082) — 把节点本地资源(非设备)也纳入 DRA 表达
- [KEP-5004:DRA Extended Resources 提议 GA(#6023)](https://github.com/kubernetes/enhancements/pull/6023) — 让现有 `extended-resource` 工作负载平滑落到 DRA
- [DRA Prioritized List 标记为 implemented(#6085)](https://github.com/kubernetes/enhancements/pull/6085) — "按优先级列设备候选"已落地

启示:Derived Attributes 与 numaNode 标准属性都在解决同一类问题——多设备协同放置(GPU+NIC 共置、NUMA 对齐),这正是大模型训练/推理对网络与内存拓扑敏感的核心诉求。如果产品要支持 RDMA/高速互联场景,这两个 KEP 决定了未来在 DRA 层如何表达拓扑约束,值得现在就读。numaNode 标准化对自研加速器(如昇腾 NPU)driver 同样关键:numa 属性要按这个命名上报,才能与 NVIDIA/AMD 设备在同一 matchAttribute 体系里互通。

两个仓库窗口内都很安静,只有依赖升级:[LWS #856](https://github.com/kubernetes-sigs/lws/pull/856) 升 k8s 0.36;JobSet 三个 dependabot bump(含 [k8s group 8 项更新 #1236](https://github.com/kubernetes-sigs/jobset/pull/1236))。无功能变更。

窗口前补记 —— [JobSet v0.12.0(2026-05-08 发布)](https://github.com/kubernetes-sigs/jobset/releases/tag/v0.12.0) 是上一份周报后的第一个 release,值得回看:

启示:JobSet v0.12.0 的 Elastic JobSets + RestartJob 系列,配合 Kueue ElasticJobsViaWorkloadSlices 走 Beta,弹性训练"加 pod 不重建 Job"的全链路在 JobSet 0.12 + Kueue v0.18 基本闭合。RestartJob 系列重启动作是训练故障自愈的细粒度开关,做训练平台可直接复用,不必自造重启逻辑。

## KEP 动向
- [KEP-4671:Gang Scheduling 冲刺 v1.37 Beta(#6076)](https://github.com/kubernetes/enhancements/pull/6076) —— 本周提交。除 beta 毕业外还提议:PodGroup/PodGroupTemplate 的 minCount 可变(弹性训练可在线调最小并发数)、kube-scheduler 新增 PlacementFeasible 扩展点检查 PodGroup 可行性、用 workload-aware preemption 完全取代 delayed preemption、三个 feature gate 合并为 GenericWorkload
- [新增 KEP-6089:Workload Aware Scheduling Controller APIs(#6092,2026-05-18 初稿)](https://github.com/kubernetes/enhancements/pull/6092) —— 与 Gang/Workload API 同源的"工作负载感知调度"控制器 API,是 gang 之后调度抽象的下一步,值得盯
- DRA 系列(6080 / 6072 / 5677 / 5517 / 5004 / 6085)见上节 DRA 小节

启示:KEP-4671 把三个 gate 合并成 GenericWorkload、并新增 kube-scheduler PlacementFeasible 扩展点,意味着 gang / workload-aware 调度要"长进 kube-scheduler 本体",而不是只做外挂插件。我们若仍用 Volcano / scheduler-plugins coscheduling 做 gang,1.37 之后要评估迁移到上游原生路径——长期产品里别再自研 PodGroup CRD。minCount 可变是弹性训练的关键拼图:gang 的最小并发门槛能在线调整,而不必重建作业。
