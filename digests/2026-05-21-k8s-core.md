# K8s 自身核心周报 2026-05-21

窗口:2026-05-14 → 2026-05-21(7 天)

## 摘要(3 条以内)

- **KEP 大周,1.37 周期全面铺开**:本窗口 enhancements 仓动作密集,呈两条线——一边把 1.36 已落地的特性批量标 `implemented`(Mutating Admission Policies KEP-3962 stable、Mutable CSINode Allocatable KEP-4876 stable、OCI Artifact VolumeSource KEP-4639 stable),一边给 1.37 排晋级目标(metrics.k8s.io → Stable KEP-5207、Node Declared Features → GA KEP-5328、object/external metrics 缩容到零 → beta KEP-2021)。版本节奏判断的最佳信号都在这一批里。
- **VPA InPlace updateMode 正式合并**:[autoscaler#8888](https://github.com/kubernetes/autoscaler/pull/8888) 于 2026-05-14 合入,VPA 第一次有了原生的 in-place 更新模式实现;叠加上游 KEP-5836(In-Place Resize 抢占语义)持续推进,VPA "改资源不重建 Pod" 的生产化路线从提案进入实质代码阶段。
- **Runtime 安全维护周**:containerd 四条维护分支同日齐发补丁([v2.3.1](https://github.com/containerd/containerd/releases/tag/v2.3.1) / v2.2.4 / v2.0.9 / v1.7.32),修 CVE-2026-46680,并把默认 seccomp 策略收紧(屏蔽 `AF_ALG`)、user namespace 下禁用 overlayfs rebase——企业升级时这批补丁要尽快吃。

## 调度(kube-scheduler)

- [KEP-5598 Opportunistic batching → beta](https://github.com/kubernetes/enhancements/pull/6039) — 调度器机会式批量打分申请 1.37 beta,大集群调度吞吐优化进入 beta 阶段。

- [KEP-5869 Wildcard Matching in Toleration Keys(alpha)](https://github.com/kubernetes/enhancements/pull/5880) — toleration key 支持通配匹配,可一条 toleration 覆盖一族 taint(如 `readiness.k8s.io/*`)。

## 节点 & kubelet

本窗口 sig-node 是 KEP 产出最密集的 SIG,新提案集中在镜像、安全 profile、资源配置三块:

- [KEP-6090 kubelet-CRI exec 环境变量注入(新)](https://github.com/kubernetes/enhancements/pull/6091) — 给 `kubectl exec` 会话注入环境变量的标准化机制。
- [KEP-6061 OCI artifact 形式分发安全 profile(新)](https://github.com/kubernetes/enhancements/pull/6062) — seccomp / AppArmor profile 用 OCI artifact 分发,不再依赖把文件预置到节点磁盘。
- [KEP-6083 Digest-Aware ImagePullPolicy Defaulting(新)](https://github.com/kubernetes/enhancements/pull/6084) — 镜像引用带 digest 时,`imagePullPolicy` 默认按 `IfNotPresent` 处理(对齐既有文档),减少不必要的 registry 拉取。
- [KEP-6063 Per-Pod PID Limit 可配置(新)](https://github.com/kubernetes/enhancements/pull/6064) — 单 Pod PID 上限从节点级常量变为可配置。
- [KEP-5620 Node resizing via balloons(alpha)](https://github.com/kubernetes/enhancements/pull/5624) — 用 "balloon" Pod 占位实现节点可见资源的伸缩。
- [KEP-5714 是否 unshare cgroup namespace 可指定](https://github.com/kubernetes/enhancements/pull/5715)、[KEP-6051 CRI Streaming Authorization Hook](https://github.com/kubernetes/enhancements/pull/6052)、[KEP-4563 EvictionRequest API 更新(WIP)](https://github.com/kubernetes/enhancements/pull/6074) 持续推进。

## 自动扩缩容(VPA / CA / Karpenter)

### VPA

- **[autoscaler#8888 VPA InPlace updateMode 合并](https://github.com/kubernetes/autoscaler/pull/8888)**(2026-05-14)— VPA 第一次有了原生 in-place 更新模式,可在不重建 Pod 的前提下调整 request/limit。这是 VPA 长期"准 alpha"困境的关键一跃:配合上游 KEP-1287(In-Place Pod Resize)与 KEP-5836(抢占语义),VPA 真正生产化的拼图正在补齐。
  - 启示:我们做"AI 工作负载自动调资源"可以开始拿 VPA InPlace updateMode 在测试集群验证——尤其训练/推理 Pod 重建代价高,in-place 调整价值最大。
- VPA 加 `MemoryAggregationInterval` 的 Seconds/Count 逐 VPA 配置项;[#9613 VPA E2E 测试迁移到 Helm](https://github.com/kubernetes/autoscaler/pull/9613)。

### Cluster Autoscaler

- [#9598 基于 ResourceQuota 的缩容抑制](https://github.com/kubernetes/autoscaler/pull/9598) — 缩容逻辑改用 resourcequotas 包作为权威来源,cloud provider 侧可更灵活实现缩容抑制;同期还加了"按 ResourceQuota 平衡相似 node group"与 CapacityQuota API 的晋级标准。
  - 启示:CA 正把 ResourceQuota 纳入扩缩容决策闭环——多租户集群里给租户配额能直接约束节点弹性行为,对我们做企业级多租户资源治理是个可用抓手。
- [#9514 Azure VMSS 实现 AtomicIncreaseSize](https://github.com/kubernetes/autoscaler/pull/9514) — Azure 节点池原子扩容。
- 其他:Scale-up salvo(单轮 CA 循环内多次扩容)、informer 缓存剥离 managedFields 降内存、把带 nominated node name 的 Pod 写入 cluster snapshot、drop 对 `k8s.io/kubernetes` 的直接依赖。

### Karpenter

- [#3025 调度时忽略 Node Readiness Controller 的 taint](https://github.com/kubernetes-sigs/karpenter/pull/3025) — Node Readiness Controller(1.36 beta)会打 `readiness.k8s.io/<rule>` 前缀 taint,Karpenter 不再把它当作需要新建节点的信号,避免误扩容。
- [#3014 DRA 设备分配追踪](https://github.com/kubernetes-sigs/karpenter/pull/3014) — Karpenter 开始在扩缩容决策里跟踪 DRA 设备分配(DRA 设备层本身归 `k8s-ai-infra`,此处仅记 Karpenter 侧能力)。
- [#3000 按 zone 统计 pending pod 指标](https://github.com/kubernetes-sigs/karpenter/pull/3000)、[#2743 卷拓扑调度的备选方案支持](https://github.com/kubernetes-sigs/karpenter/pull/2743)。
