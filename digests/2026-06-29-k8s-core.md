# K8s 自身核心周报 2026-06-29

窗口:2026-06-22 -> 2026-06-29(7 天)

## 摘要(3 条以内)
- [Kubernetes v1.37.0-alpha.2](https://github.com/kubernetes/kubernetes/releases/tag/v1.37.0-alpha.2) 已进入 v1.37 alpha 验证窗口；本周需要关注 sig-node、sig-scheduling、sig-api-machinery、sig-autoscaling 的 KEP 状态变化。
- [Gateway API v1.6.0-rc.2](https://github.com/kubernetes-sigs/gateway-api/releases/tag/v1.6.0-rc.2) 引入 HTTPS listener 必须配置 TLS 的 breaking change，并继续推进 Telemetry API、ListenerSet、GRPCRoute/HTTPRoute conformance。
- Cluster API、Calico、Gatekeeper、Operator SDK 本周都有 release 信号，底座升级风险集中在 Gateway TLS、安全策略、CAPI K8s 1.36 支持、operator 工具链依赖。

## K8s 核心版本节奏

- [Kubernetes v1.37.0-alpha.2](https://github.com/kubernetes/kubernetes/releases/tag/v1.37.0-alpha.2) — v1.37 周期 alpha release。
  - 启示:v1.37 已经进入可跟踪窗口。我们应建立 KEP watchlist，特别是 DRA、in-place resize、Topology Manager、API machinery、autoscaling 相关状态，避免到 beta/RC 才发现产品 API 受影响。
- [Cluster API v1.13.3](https://github.com/kubernetes-sigs/cluster-api/releases/tag/v1.13.3) — 管理集群支持 Kubernetes v1.32-v1.36，workload 集群支持 v1.30-v1.36；安全修复包含 OwnerReferencesPermissionEnforcement admission controller RBAC。
  - 启示:如果我们的集群生命周期能力对接 CAPI，K8s 1.36 兼容性和 owner reference admission 权限要纳入升级检查，避免 operator 创建从属资源失败。
- [Cluster API v1.14.0-alpha.0](https://github.com/kubernetes-sigs/cluster-api/releases/tag/v1.14.0-alpha.0) — alpha 版本引入 API module 拆分、controller-runtime v0.24/controller-tools v0.21、CAPD 向 Dev backend 迁移。
  - 启示:CAPI 正在重构 API/测试/依赖边界。我们的 CAPI provider 或 clusterctl 集成应关注 v1.14 breaking changes，但生产基线仍应先停在 stable patch。

## 调度(kube-scheduler)

- [KEP-5598 Rescoring flow](https://github.com/kubernetes/enhancements/pull/6212) 本周更新 PreScore requirement 描述。
  - 启示:Rescoring 会影响调度插件在二次评分中的数据准备方式。我们如果做自定义 scheduler plugin，要确认 PreScore/Score/Reserve 等扩展点在新 flow 下的行为。
- 通用调度层无新的 kube-scheduler release note；AI 专用 DRA/Kueue 内容已放在 `k8s-ai-infra`，本周这里只保留对调度框架语义的跟踪。

## 节点 & kubelet

- [KEP-5972 Dynamic Containers](https://github.com/kubernetes/enhancements/pull/6169) 与 [KEP-6007 Topology Manager single-numa-node density](https://github.com/kubernetes/enhancements/pull/6008) 本周有更新。
  - 启示:Dynamic Containers 如果推进，会改变 Pod 内运行时变更能力；Topology Manager 选项会影响高密度节点工作负载放置。AI 平台需要关注这两者对 sidecar/agent 注入、NUMA 亲和和节点池 profile 的影响。

## 自动扩缩容(VPA / CA / Karpenter)

- [KEP-3953 In-place Node Resource Resize](https://github.com/kubernetes/enhancements/pull/3955) 覆盖 sig-node/sig-scheduling/sig-autoscaling 等多个 SIG。
  - 启示:节点资源在线调整会影响 autoscaler 与 scheduler cache 对 capacity/allocatable 的判断。对 AI 节点池来说，它可能与设备资源、拓扑资源、quota 产生交互，不能只按通用 CPU/Mem 场景评估。
- [Kueue v0.18.2](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.18.2) 修复 MultiKueue manager cluster admitted Pod 误触发 Cluster Autoscaler。
  - 启示:即使这是 AI 侧组件问题，底层教训是 autoscaler 信号必须区分真实 Pod 和控制面占位对象。我们的多集群或队列产品要避免 Pending/Admitted 状态误导扩容。

## Runtime(containerd / CRI-O / runc)

- 本窗口未确认到 containerd / CRI-O / runc 与 AI 平台直接相关的重大 release。维持 CVE、runtime handler、cgroup v2 和 image GC 常规观察即可。

## API Machinery & Operator 生态

- [Operator SDK v1.42.3](https://github.com/operator-framework/operator-sdk/releases/tag/v1.42.3) — 主要是依赖升级，包括 go-jose、OpenTelemetry、go-git、distribution 等。
  - 启示:operator 工具链 patch 多数不是功能性变化，但会影响安全扫描和生成物。我们的 operator 模板升级时要重跑 bundle/manifests diff，避免 generated files 漂移。
- [Cluster API v1.14.0-alpha.0](https://github.com/kubernetes-sigs/cluster-api/releases/tag/v1.14.0-alpha.0) 升级 controller-runtime v0.24/controller-tools v0.21。
  - 启示:controller-runtime v0.24 正成为 K8s 1.36 生态基线。自研 operator 升级前要重点回归 SSA、cache、webhook、fakeclient 单测差异。

## 网络 & 存储(大盘)

- [Gateway API v1.6.0-rc.2](https://github.com/kubernetes-sigs/gateway-api/releases/tag/v1.6.0-rc.2) — breaking change: HTTPS protocol listener 必须配置 TLS；同时推进 Telemetry API、ListenerSet、TLSRoute hostname limit、GRPCRoute conformance。
  - 启示:MaaS/LLM endpoint 的网关配置不能假设 HTTPS listener 可后补 TLS。我们的安装器和 preflight 应直接校验 Gateway listener TLS，否则升级后可能路由不可用或不合规。
- [Calico v3.32.1](https://github.com/projectcalico/calico/releases/tag/v3.32.1) — 发布容器镜像、binaries、Kubernetes manifests、Windows 包、OpenShift manifest bundle。
  - 启示:Calico 仍是企业客户常见 CNI。AI 服务网关、跨节点 RDMA/普通网络策略、OpenShift bundle 交付需要把 Calico patch 纳入网络兼容矩阵。

## 策略 & 安全

- [Gatekeeper v3.23.0-rc.1](https://github.com/open-policy-agent/gatekeeper/releases/tag/v3.23.0-rc.1) — 新增 structured log、remote cluster mode status resource routing、mutation ApplyTo operations 字段；同时修复 VAP API discovery 重试、OCI pulls 迁移到 oras-go v2 等。
  - 启示:策略引擎正在同时覆盖多集群、VAP、OCI 分发和 mutation。AI 平台若要做企业合规，应把模型服务、GPU/NPU 节点、MaaS endpoint 的策略模板纳入 Gatekeeper/Kyverno 兼容验证。

## KEP 动向(非 AI)

- [KEP-6061 OCI artifact-based security profile distribution](https://github.com/kubernetes/enhancements/pull/6062) — 安全 profile 通过 OCI artifact 分发。
  - 启示:如果这条线成熟，seccomp/AppArmor/安全 profile 可以像镜像一样被版本化和分发。AI workload 的安全 profile、GPU/NPU runtime 约束有机会进入统一制品链。
- [KEP-6116 EndpointSelector](https://github.com/kubernetes/enhancements/pull/6127) — 选择 Pod endpoints 不再强依赖 Service。
  - 启示:对网关、服务网格和推理路由来说，这可能减少“为了选 endpoint 必须创建 Service”的耦合。后续可关注其与 Gateway API / InferencePool 的关系。
