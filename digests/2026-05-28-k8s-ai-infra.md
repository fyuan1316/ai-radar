# K8s AI 基础设施周报 2026-05-28

窗口:2026-05-21 → 2026-05-28(7 天)

## 摘要(3 条以内)
- **DRA Extended Resources 升 GA 已合入 1.37**:KEP-5004([#6023](https://github.com/kubernetes/enhancements/pull/6023),2026-05-27 merged)把现有 `nvidia.com/gpu` 这类扩展资源直接路由到 DRA driver,无需改 Pod spec —— 这是把所有存量 GPU 工作负载平滑迁到 DRA 体系的关键拼图,1.37 之后扩展资源是 DRA 的"前端"而不是平行方案。
- **Kueue v0.18 把 DRA 配额闭环再加一道安全网**:KEP-2941 partitionable devices 支持([#10283](https://github.com/kubernetes-sigs/kueue/pull/10283))+ 事件驱动 DeviceClass 跟踪([#11552](https://github.com/kubernetes-sigs/kueue/pull/11552))+ RejectDRAWhenDisabled feature gate([#10964](https://github.com/kubernetes-sigs/kueue/pull/10964),默认 Beta、默认开启)合入,堵住"DRA gate 关掉时 DRA workload 静默以零资源准入、绕过配额"的安全洞 —— 多租户 GPU 集群必须升 v0.18 之前确认这条路径。
- **DRA Preemption 首版 KEP 出炉、Workload-aware Preemption 冲 1.37 Beta**:新增 KEP-5690([#6113](https://github.com/kubernetes/enhancements/pull/6113),2026-05-25)首次定义 DRA 设备的抢占语义;KEP-5710 提交 1.37 Beta 更新([#6106](https://github.com/kubernetes/enhancements/pull/6106))彻底取消 delayed preemption、与 GangScheduling 合并到 GenericWorkload feature gate —— 调度层的"抢占"在 1.37 同时在设备维度和 workload 维度被重做。

## 调度 & 资源分配

### Kueue

本周 main 分支 + 双补丁线合计 merge 257 个 PR(其中绝大多数是补丁线 cherry-pick、`ginkgo.Ordered` 测试重构、整理 e2e 分片)。功能性变更集中在 v0.18 GA 冲刺:

DRA / 设备配额(本周主线):
- [#10283 KEP-2941:Partitionable devices 支持](https://github.com/kubernetes-sigs/kueue/pull/10283) — Kueue 正式认识 DRA partitionable devices,把"一张 GPU 切多份"映射到 ResourceFlavor 配额。配合 [#11552 / #10973 事件驱动 DeviceClass 跟踪](https://github.com/kubernetes-sigs/kueue/pull/11552),DeviceClass 增删不再需要重启 Kueue 重新建索引。
- [#10964 RejectDRAWhenDisabled](https://github.com/kubernetes-sigs/kueue/pull/10964) — 新增同名 feature gate(Beta、默认开启)。若集群关掉了 `DynamicResourceAllocation` gate 而 workload 又用了 ResourceClaim,以前会被静默以零资源准入、彻底绕过配额(DRA scheduler 直接分配设备),现在会在 webhook 与 reconcile 两处一并拒绝,标记 `WorkloadInadmissible`。**安全级修复**,多租户 GPU 集群升级前要确认这条路径。
- [#11405 Skip DeviceClass index if DRA API is not available](https://github.com/kubernetes-sigs/kueue/pull/11405) — Kueue 启动时如果集群没装 DRA CRD/API,跳过 DeviceClass 索引,不再 panic。

弹性 / 优先级 / 配额上限:
- [#11547 ElasticJobsViaWorkloadSlices → Beta](https://github.com/kubernetes-sigs/kueue/pull/11547) — 上周补的 webhook 校验合入后正式 Beta 毕业,v0.18 默认开启。配合 JobSet v0.12 的 Elastic JobSets 与 RestartJob,弹性训练"加 pod 不重建 Job"全链路在 1.37 周期闭合。
- [#10798 WorkloadPriorityClassDefaulting](https://github.com/kubernetes-sigs/kueue/pull/10798) — KEP-10765 的实现版本合入。给没带 `kueue.x-k8s.io/priority-class` 标签的 workload 自动套用名为 `default` 的 WorkloadPriorityClass,把"pod 调度优先级"与"排队优先级"两件事彻底切开。沿用上周 LocalQueueDefaulting 的约定式模式。
- [#11388 Workload PodSet 上限提到 16](https://github.com/kubernetes-sigs/kueue/pull/11388) — 由 8 提到 16(release-note 与 PR 标题口径有出入,以 16 为准)。直接影响 Ray:RayCluster / RayJob / RayService 最多可有 15 个 worker group(留 1 个给 head)。我们若做大型 Ray 集群产品,这是上限松动的关键节点。
- [#11236 RetainFirstAdmission 迁移模式](https://github.com/kubernetes-sigs/kueue/pull/11236) — 给 concurrent admission 迁移加一个"把 Workload 锁定到首次准入 flavor"的过渡开关,对存量队列升级时的可预测性有用。
- [#11156 Amount struct 处理配额溢出](https://github.com/kubernetes-sigs/kueue/pull/11156) — 上周 #11140 / #11151 修补丁线整数溢出击穿配额的根因清理,把所有 quota 数值过一遍 `Amount` 类型,杜绝再出现同类越界。

MultiKueue 稳定性:
- [#11668 + #11693 + #11692 远端 cluster 正在连接时 requeue workload](https://github.com/kubernetes-sigs/kueue/pull/11668) — 修上周 #11207 / #11304 / #11305 之后留下的最后一类挂死 case,补丁线 0.16/0.17 同步打。
- [#11559 / #11497 fix MultiKueue 重排 / 重连延迟问题](https://github.com/kubernetes-sigs/kueue/pull/11559) — `AllAtOnce` dispatcher 不再因为 `nominatedClusterNames` 顺序抖动反复更新 workload,失联恢复也更快。

性能 & 杂项:
- [#11422 大规模调度器阈值用 CI 数据重标](https://github.com/kubernetes-sigs/kueue/pull/11422) — 性能调参,无 API 变化。
- [#11399 把 effective requests 纳入调度等价 hash](https://github.com/kubernetes-sigs/kueue/pull/11399) — 修因 hash 撞车把不等价 workload 当作等价的 bug,影响公平性。
- [#11561 修 concurrent admission cache 空指针 panic](https://github.com/kubernetes-sigs/kueue/pull/11561) — 稳定性补丁。
- [#11628 用 LocalQueue 引用做 inflight pending 计数](https://github.com/kubernetes-sigs/kueue/pull/11628) — 内部数据结构整理。

启示:**v0.18 GA 临门一脚,DRA 路径默认开启 + 默认拒绝"DRA gate 关闭却用 ResourceClaim 的 workload"是这版的安全护栏**,而非锦上添花。我们做 GPU 多租户配额的产品要把升级前置条件写清楚:(1) Kubernetes ≥1.36 且 DRA gate 开启;(2) RejectDRAWhenDisabled gate 显式确认;(3) 灰度 v0.18 时把 partitionable devices 路径单独跑一遍 e2e。PodSet 上限 8→16 对支持大型 Ray 拓扑(>8 worker group)是硬性解锁,产品文档可同步更新。

### DRA(Dynamic Resource Allocation)

> 任务原列的 `kubernetes-sigs/dra-evolution` 仓库已 404,DRA 走 k/enhancements。本周 KEP 动静很大,GA / Alpha 双线推进。

GA / 收敛动作:
- [KEP-5004 DRA Extended Resources 升 GA(#6023,2026-05-27 merged)](https://github.com/kubernetes/enhancements/pull/6023) — issue [#5004](https://github.com/kubernetes/enhancements/issues/5004) 已带 `stage/stable` 标签。让现有 `nvidia.com/gpu` / `amd.com/gpu` 这类扩展资源的工作负载,经过 DRA driver 转发后能直接被 DRA 体系核算。这是 1.37 周期内"扩展资源不再是 DRA 的平行方案,而是 DRA 的兼容前端"这一战略落地的关键 PR。
- [KEP-5055 DRA Device Taints and Tolerations 升 GA(#6040)](https://github.com/kubernetes/enhancements/pull/6040) — 设备级的 taint/toleration 走向 GA,无 API 变更。配合 DRA 的设备健康/维护流程,可以让"故障 GPU 自动从可调度池中剔除"成为标准能力。

新增 KEP:
- [KEP-5690 DRA Preemption(#6113,2026-05-25 新建)](https://github.com/kubernetes/enhancements/pull/6113) — 首份 DRA 抢占设计。当前 DRA 设备分配后无任何抢占语义,长跑训练把卡占满后无法被推理任务抢回。issue [#5690](https://github.com/kubernetes/enhancements/issues/5690) 标签 `stage/alpha`,`sig/scheduling`、`wg/device-management` 联合。我们如果要做"分级训练/推理共池",这是决定能不能做的根 KEP,值得贴身跟。
- 上周新增的 [KEP-6080 Derived Attributes(#6081)](https://github.com/kubernetes/enhancements/pull/6081)、[KEP-6072 numaNode 标准属性(#6073)](https://github.com/kubernetes/enhancements/pull/6073) 本周持续讨论,无状态变化。

仍在推进的:
- [KEP-5677 partitionable + consumable devices 二次 Alpha(#6068)](https://github.com/kubernetes/enhancements/pull/6068)、[KEP-5517 Node Allocatable Resources alpha2(#6082)](https://github.com/kubernetes/enhancements/pull/6082)、[KEP-5963 Device Compatibility Groups(#5964)](https://github.com/kubernetes/enhancements/pull/5964)、[KEP-5981 Sharing Affinity for Consumable Capacity(#5987)](https://github.com/kubernetes/enhancements/pull/5987) 本周都在持续 review,#5987 本周补 PRR 审批文件([#6119](https://github.com/kubernetes/enhancements/pull/6119)),进入 production-readiness 评估。

启示:**1.37 周期里 DRA 同时打出三张牌 —— 兼容(KEP-5004 把存量扩展资源吸过来)、抢占(KEP-5690 第一次能从 DRA 体系里把卡夺回来)、设备级运维(KEP-5055 taint/toleration GA)**。我们做 AI 基础设施的产品决策要据此调整:已有的"扩展资源 + 自研抢占器"路径在 1.37 之后失去优势,应该把统一调度入口收到 DRA。自研 NPU/加速器 driver 现在就要决定:是否在 1.37 周期跟进 DRA driver 化,而非继续维持 device-plugin 形态。

启示:**1.37 是 K8s 调度抽象重做的关键版本** —— Gang(workload-level) + Workload-aware Preemption(workload-level)+ DRA Preemption(device-level)+ CompositePodGroup(hierarchical)四条线在同一发布窗口里同时推进。我们若仍在自研 PodGroup CRD / 自研抢占器,本周就要起草迁移评估:1.37 之后这条上游路径会被默认认作"正路",自研路径长期可能拿不到生态(YAML、Helm chart、operator)。CompositePodGroup 是新冒头的方向,值得跟,但不必现在投入。
