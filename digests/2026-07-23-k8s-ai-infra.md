## 调度 & 资源分配

### Kueue(v0.19.0,本窗口重头)

发布链:v0.19.0-rc.0(07-21)→ v0.19.0 正式版 + v0.18.4 + v0.17.8(07-22 三分支齐发)。https://github.com/kubernetes-sigs/kueue/releases/tag/v0.19.0

升级前必读(硬约束):
- **`WaitForPodsReady` 默认开启**:新装与未显式配置的老装都会启用默认配置(30 分钟 timeout / 30 分钟 recovery)。同时引入临时开关 `DisableWaitForPodsReady` 可退回。(#11855 / #13107)
- **MultiKueue 安全修复**:`locationType=Path` kubeconfig 处理漏洞,有 `MultiKueueCluster` 写权限者可诱导 controller 读任意文件;现强制路径落在 `/etc/multikueue/kubeconfigs` 下。用 Path 模式的需迁移文件或改用 `locationType=Secret`/`ClusterProfile`。(#12223)
- **RayCluster/RayJob 配额回填**:autoscaler sidecar(默认 500m/512Mi)与 SidecarMode submitter sidecar(500m/200Mi)现计入 head PodSet 配额,升级后相关 ClusterQueue 需相应加配额,否则 admission 失败。(#12405 / #12454)

功能亮点(与 AI 调度强相关):
- **DRA 整合两 gate 晋级 Beta 默认开**:`KueueDRAIntegrationExtendedResource`(#13102)、`KueueDRAIntegrationPartitionableDevices`(#13167);并删掉旧 `DynamicResourceAllocation` gate,统一走 `KueueDRAIntegration`(#12258)。Partitionable Devices 支持同一 DeviceClass 出现在多个 `deviceClassMappings` 用不同 counter 源做多计数追踪(#13018),新增基于容量的多分配 DRA 配额(#13152)。本窗口另有一大批 DRA 配额精确性修复(int64 溢出钳制 #12909/#12945/#12897、DeviceClass 删除后 stale 状态/requeue #11927/#11929、matchAttribute 误拒 #12451、nil 解引用防护 #12889)。→ Kueue 的 DRA 支持这一版从 alpha 打磨进 Beta 阶段,是配额层对接 GPU 分片/共享的关键成熟度信号。
- **TAS(拓扑感知调度)**:`TASMultiLayerTopology` 晋级 Beta 默认开(每 workload 配多层 slice 拓扑约束,#12290);新增 `TASAssignmentsEncodingByHostnamePrefix`(Beta 默认开)hostname 前缀编码,支持 TAS 覆盖 10 万+ 节点规模(#11579)。
- **WorkloadAwareScheduler**:新增 `SchedulerLibraryIntegration` gate,用上游 `scheduler-library` 做节点域 TAS fit 检查(#13261)——与下面 KEP-5732/5547 的"workload 感知调度"上游主线对齐。
- 其他:每 Workload 最大 PodSet 从 10 提到 18(#12819)、OOTB QPS/并发上调(QPS 300 / Burst 500,#12440)、一批 MultiKueue 可观测指标(`multikueue_workloads_admitted_total` 等)与未准入工作负载可观测性(`UnadmittedWorkloadsObservability`)。

### DRA

- **本窗口 DRA 的实质推进主要发生在 Kueue 侧(见上)**,而非 `kubernetes-sigs/dra-evolution`(该仓窗口内无提交)。这延续了近期趋势:DRA 核心 API 稳定后,竞争焦点转移到"消费方"(队列/调度器)如何把 DRA 设备纳入配额与拓扑决策。
- 值得关注:Kueue 的 Partitionable Devices 多计数模型(#13018)+ 容量配额(#13152),是把 DRA 分片设备做进配额账本的早期实践,对我们做 GPU 细粒度共享的配额设计有直接参考价值。

### LWS / JobSet

- **LWS DisaggregatedSet 放置策略(KEP-848)落地**:`#916` 实现 `spec.placementPolicy`,controller 在 create 时向被管 LWS pod template 注入亲和/反亲和——`ExclusiveSlice`(slice 内角色同域 + 本 set 的 slice 跨域打散)、`ExclusiveTopology`(再加"跨所有 DisaggregatedSet 每域一个 slice");放置刻意不进 revision hash。配套 `#906` 合入 KEP 设计文档、`#912` webhook 拒绝 `maxSurge=0+maxUnavailable=0`、`#896` 把 LWS metadata 透传到 StatefulSet。→ P/D 分离推理在原生 K8s(非 Volcano/自研)侧的编排原语正在成型,值得对标。https://github.com/kubernetes-sigs/lws/pull/916
- **JobSet DRA 集成文档(#1277)**:记录如何通过 `DRAWorkloadResourceClaims` gate 用 PodGroup 级 ResourceClaims 配合 JobSet 做 workload-aware 调度(基于 dra-example-driver 在 Kind 上验证)。另 `#1269` 修排他放置在 follower pod 不匹配时的修复逻辑、`#1279` 拒绝负数 replicated job replicas。https://github.com/kubernetes-sigs/jobset/pull/1277
- scheduler-plugins(gang)本窗口无 release/合入。

## KEP 动向

- **KEP-5732 Topology-aware workload scheduling → Beta,latest-milestone v1.37**(alpha v1.36 → beta v1.37,计划 v1.39 GA)。把整个 workload 当作一个拓扑约束单元交给 kube-scheduler;本窗口 `#6234` 重命名指标 `scheduler_generated_placements_total`。这是"workload 感知调度"上游主线,直接影响多节点训练/推理放置未来是否还需要 Volcano/自研 gang。https://github.com/kubernetes/enhancements/tree/master/keps/sig-scheduling/5732-topology-aware-workload-scheduling
- **KEP-5547 Integrate Workload APIs with Job Controller(sig-apps,alpha,v1.37)**:把 Workload API 接进 Job Controller;`#6243` 按实现进展更新 KEP。与 5732 是同一"Workload"抽象的调度侧与控制器侧两条腿。https://github.com/kubernetes/enhancements/tree/master/keps/sig-apps/5547-integrate-workload-with-job
- **KEP-5598 Opportunistic batching(sig-scheduling,Beta,latest-milestone v1.37,计划 v1.38 GA)**:`#6212` 澄清 Rescoring 流程里的 PreScore 要求。调度吞吐优化,对高并发 pod 提交(大批训练任务)有益。https://github.com/kubernetes/enhancements/tree/master/keps/sig-scheduling/5598-opportunistic-batching
- **KEP-848 DisaggregatedSet(LWS)**:非 k/enhancements 而在 LWS 仓推进(见上),本窗口从设计文档走到代码落地,是 disaggregated inference 的原生编排 KEP。https://github.com/kubernetes-sigs/lws/pull/906
- (交叉参考)v1.37.0-beta.0 已切出,in-tree `WorkloadAwarePreemption` 新增 `PodGroupPostFilter` 扩展点 + `CompositePodGroup v1alpha3`,详见同日 k8s-core digest。
