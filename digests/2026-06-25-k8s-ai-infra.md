# K8s AI 基础设施周报 2026-06-25

> 窗口:2026-06-18 ~ 06-25。数据源见 `tasks/k8s-ai-infra.md`。本期信号集中在 **DRA 的"质性共享"演进**(KEP-5981)与 **Kueue 对 DRA 配额计数的打通**——两者合起来,正是把"GPU/RDMA 分片 + 队列配额"从演示推向多租户可用的关键拼图。

## 摘要(3 条)

1. **DRA 从"数量共享"迈向"模态门控"**:新 KEP-5981(DRA Sharing Affinity)让调度器能感知部分分配设备被"锁定"到某种配置模式(子网/分区键/bitstream),只把兼容工作负载排进剩余容量——这是多租户 GPU/FPGA/RDMA 共享的前提,且显式废弃了现网的 driver 端 placeholder 套路。 https://github.com/kubernetes/enhancements/pull/5987
2. **Kueue 打通 DRA 配额计数对 `matchAttribute` 的支持**(#12451,已 cherry-pick 到 release-0.18):配额计数不再因设备约束/配置而拒绝工作负载,放行了 GPU+NIC 经 `pcieRoot` 共置这类拓扑亲和场景。 https://github.com/kubernetes-sigs/kueue/pull/12451
3. **NVIDIA k8s-device-plugin v0.19.3 回退了 mofed/gdrcopy 默认启用**:GPUDirect RDMA / GDRCopy 相关特征恢复为默认关闭,升级前需确认是否依赖其默认开启。 https://github.com/NVIDIA/k8s-device-plugin/releases/tag/v0.19.3

## GPU / 加速器管理

- **NVIDIA/k8s-device-plugin v0.19.3**(06-22 发布):补丁版。最值得注意的是 **revert 了 mofed、gdrcopy 的默认启用**(#1837),即 GPUDirect RDMA / GDRCopy 特征探测恢复默认关闭;其余为 Go 1.26.4、golang.org/x/net v0.56.0、distroless 基础镜像等例行 bump。若你的栈依赖这几个特征"开箱即开",升级后需显式打开。 https://github.com/NVIDIA/k8s-device-plugin/releases/tag/v0.19.3
- **NVIDIA/gpu-operator v26.3.3**(release-26.3 分支):补丁发布。新增 **OCP 4.22 作为受支持版本**(OLM,#2578),runc 升 1.4.3(#2568),filepath-securejoin 0.7.0(#2560,安全相关路径处理)。对标 OAI 时可留意其对新 OpenShift 版本的支持节奏。 https://github.com/NVIDIA/gpu-operator/pull/2578
- **NFD:x86-64 微架构等级(v1–v4)CPUID 标签**(#2490,已合):节点会打出 `feature.node.kubernetes.io/cpu-cpuid.X86_64_V3=true` 等累积标签,可用 `nodeSelector` 按"最低微架构等级"调度,而无需数值比较。对需要 AVX-512/AVX2 的推理/向量化算子选型节点很实用。 https://github.com/kubernetes-sigs/node-feature-discovery/pull/2490
  - 同期 NFD 还合入:不完整 cache sync 时防止误删 label(#2415,稳定性)、worker 滚动更新默认 maxUnavailable 10%(#2519)、runc 升至 1.5.0(#2520)。

## KEP 动向

本期最有价值的信号,均为 **SIG-Scheduling / WG-Device-Management** 方向、且都直接关乎加速器:

- **KEP-5981:DRA Sharing Affinity for Consumable Capacity**(#5987,open,06-22 更新)。在 KEP-5075(Consumable Capacity,数量共享)之上加一层**质性门控**:driver 通过 ResourceSlice 上的 `Device.SharingAffinity` 声明哪些属性键约束共享;调度器在 Filter 阶段从已绑定 ResourceClaim 推导设备的"模态锁"状态(如 RDMA NIC 的子网、FPGA 的 bitstream、多租户 GPU 的分区键),只把**配置兼容**的工作负载排进设备剩余容量,Reserve 时即时上锁(同批次后续调度周期立即可见,无需 informer 往返)。任何歧义降级为 `Unknown` 并排除该设备直至排空。配套两个特征门控:`DRAStructuredDeviceConfiguration`(API 面)+ `DRASharingAffinity`(行为),同 alpha 落地。**Alpha 仅保证正确性**(打包/跨节点评分/破锁抢占留到 beta)。 https://github.com/kubernetes/enhancements/pull/5987
  - **对我们产品的启示**:这是"多租户共享一块物理 GPU/NIC"从演示走向安全可用的基础设施级前提。若我们的 GPU 共享/分片方案现在靠 driver 端 placeholder ResourceSlice 兜底,该 KEP 一旦落地会让调度器成为兼容性权威、去掉数据路径上的 per-pod ResourceSlice churn——值得现在就评估迁移路径。

- **KEP-5690:DRA Preemption**(#6113,open,06-19 更新)。为 DRA 资源引入抢占语义(提案正在补全)。当前 DRA 工作负载无法像 CPU/内存那样被高优先级任务抢占设备,这是生产化的明显短板。**与 `k8s-core` 本期记录的同名条目重叠,按边界归本任务为主线跟踪。** https://github.com/kubernetes/enhancements/pull/6113

- **KEP-6007:Topology Manager 单 NUMA 节点的工作负载密度优化**(#6008,open,06-24 更新)。新增 "prefer-most-allocated NUMA node" 选项,把 pod 尽量塞进同一已分配 NUMA 域以提高密度(配套 kubelet PR k/k#138321)。对 GPU+NIC+CPU 需 NUMA 对齐的推理/训练 pod 的装箱效率有直接影响。 https://github.com/kubernetes/enhancements/pull/6008

- **KEP-5980:TopologySpreadConstraint 排除全污点域**(#6210,新建 06-19)。当 `nodeTaintsPolicy=Honor` 且某拓扑域全部节点被污点时,该域计入 `minDomains` 但不计入 skew,避免整可用区故障时阻塞调度。对跨 AZ 的训练/推理副本分布有韧性意义。 https://github.com/kubernetes/enhancements/pull/6210
