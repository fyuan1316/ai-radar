# K8s AI 基础设施周报 2026-06-18

> 窗口:2026-06-11 ~ 2026-06-18。只覆盖 AI-specific 组件(gpu-operator / NFD / DRA 设备层 / LWS / JobSet / Kueue / scheduler-plugins / KEP),通用集群能力见 `k8s-core`。

## 摘要(3 条以内)

1. **K8s 1.37 是一个"DRA + 工作负载感知调度(WAS)"的超级版本**:本周 kubernetes/enhancements 一次性合入 10+ 个面向 1.37 的 DRA KEP(可分区/可消费设备二次 alpha、ClusterResourceClaimTemplate、共享可消费容量、numaNode 标准属性、可发现设备元数据升 beta……)和一整套 WAS KEP(CompositePodGroup 原生 gang、拓扑感知调度升 beta、工作负载感知抢占升 beta)。这决定我们 6 个月后的加速器分配与 gang 调度选型方向。
2. **LWS v0.9.0 落地 DisaggregatedSet(KEP-766)**:把 prefill/decode 解耦(disaggregated serving)做成原生 API,是 LWS 从"多节点推理副本"走向"分角色推理拓扑"的关键一步;同时 RecreateGroupAfterStart 转正为 API、新增 TLS、observedGeneration。OAI 在用 LWS,这条要重点跟。
3. **Kueue 已开始消费 DRA 可分区设备**:v0.18.1 修了一批 DRA admission/requeue 的 bug,并合入"Partitionable Devices" E2E;同时**移除了已废弃的 `DynamicResourceAllocation` feature gate**——DRA 在 Kueue 侧从实验位转为常驻能力。

## GPU / 加速器管理

- **gpu-operator**:本周无新版本(最新仍 v26.3.2)。核心 PR 是 [#2518 add support for default nvidiadriver and migration from clusterpolicy to nvidiadriver](https://github.com/NVIDIA/gpu-operator/pull/2518),把驱动管理从 ClusterPolicy 迁到独立 NVIDIADriver CRD —— 详见 `2026-06-18-compute-nv-watch` 的深挖,这里只做交叉锚点。其余为 golang.org/x/net v0.56.0 安全 bump、OpenShift 兼容列表更新等常规维护。
- **k8s-device-plugin**:在准备 [v0.19.3](https://github.com/NVIDIA/k8s-device-plugin/pull/1864)(bump-release + go-nvml 0.13.2 + x/net 安全 bump),尚未正式发版,无功能性变化。
- **NFD**:静默,最新仍 v0.18.3(2025-11),本周 0 合入。硬件特征发现层本周无信号。

## 调度 & 资源分配

### Kueue(v0.18.1 / v0.17.5,2026-06-12)

补丁版,主题是 **DRA 健壮性** 与 TAS(拓扑感知调度)修复:

- DRA 修复三连:
  - [#12093](https://github.com/kubernetes-sigs/kueue/pull/12093) DeviceClass 被删/`extendedResourceName` 变更时,挂起的 DRA Workload 现在会被重新入队,不再卡在过期 admission 态;
  - [#12152](https://github.com/kubernetes-sigs/kueue/pull/12152) 当 `KueueDRAIntegrationPartitionableDevices` 门控关闭时,拒绝 `deviceClassMappings[].sources` 配置,防止不支持的可分区设备配置被吞;
  - [#12057](https://github.com/kubernetes-sigs/kueue/pull/12057) 对确定性失败的 DRA 解析不再热重试(避免 hot reconcile loop),只对瞬时失败做退避重试。
- **可分区设备(Partitionable Devices)E2E 合入** [#11420](https://github.com/kubernetes-sigs/kueue/pull/11420) + [release-0.18 backport #12318](https://github.com/kubernetes-sigs/kueue/pull/12318):Kueue 开始端到端验证 DRA 可分区设备(对应 KEP-5677,见下)——这是把"一张 GPU 切片后分配给多个排队作业"做进配额体系的前置。
- **移除已废弃 `DynamicResourceAllocation` feature gate** [#12258](https://github.com/kubernetes-sigs/kueue/pull/12258):DRA 在 Kueue 侧不再是可选实验门控。
- TAS:[#12172](https://github.com/kubernetes-sigs/kueue/pull/12172) 拓扑分配失败时正确判 `NoFit`(而非误判 Fit 占了配额);[#12034](https://github.com/kubernetes-sigs/kueue/pull/12034) TAS 现在正确处理被排除/转换的资源在节点容量统计中的计数。
- MultiKueue:[#12262](https://github.com/kubernetes-sigs/kueue/pull/12262) 阻止管理集群的 Cluster Autoscaler 为 Pod workload 误扩节点。
- 新 KEP:[#11720 Improve observability of inadmissible workloads](https://github.com/kubernetes-sigs/kueue/pull/11720)——把"为什么这个作业进不来队列"做成可观测信号,排障价值高。

### DRA(本周重头,全部来自 kubernetes/enhancements,目标版本 1.37)

> dra-evolution 仓本周 0 合入;DRA 的真实进展全在主 enhancements 仓。这是本周**最有价值的信号**。

- [KEP-5677 第二次 Alpha(1.37):partitionable + consumable devices](https://github.com/kubernetes/enhancements/pull/6068) —— 可分区 + 可消费设备,**这是 device-plugin 时代"整卡独占 / MIG 静态切分"的替代路径:一张物理设备声明为可分区,运行时按需切片并作为可消费容量分配**。直接关系我们 GPU 共享方案选型。
- [KEP-5941 Shared Consumable Capacity](https://github.com/kubernetes/enhancements/pull/5942) + [KEP-5981 Sharing Affinity for Consumable Capacity](https://github.com/kubernetes/enhancements/issues/5981) —— 共享可消费容量(如显存/带宽按量分)及其亲和性,配套 5677 的细粒度共享。
- [KEP-5729 ResourceClaim Support for Workloads → Beta(1.37)](https://github.com/kubernetes/enhancements/pull/5976) + [KEP-5978 ClusterResourceClaimTemplate](https://github.com/kubernetes/enhancements/pull/6159) —— 把 ResourceClaim 从"单 Pod"提升到"工作负载/集群级模板",让 Job/LWS/JobSet 这类多 Pod 负载能统一声明设备需求,**这是 DRA 真正能服务训练/多节点推理的关键拼图**。
- [KEP-6072 Standard numaNode Device Attribute](https://github.com/kubernetes/enhancements/pull/6073) —— DRA 设备标准化 numaNode 属性,为 NUMA/拓扑对齐(GPU-NIC-CPU 同 NUMA)打基础。
- [KEP-5304 Discoverable Device Metadata API → Beta](https://github.com/kubernetes/enhancements/pull/6160)、[KEP-5963 Device Compatibility Groups](https://github.com/kubernetes/enhancements/pull/5964)、[KEP-5517 Node Allocatable Resources alpha2](https://github.com/kubernetes/enhancements/pull/6082)、[KEP-5945 Optional Node Preparation](https://github.com/kubernetes/enhancements/pull/6104)、[KEP-6081 Derived Attributes](https://github.com/kubernetes/enhancements/pull/6081)、[KEP-5004 Handle extended resource requests via DRA Driver](https://github.com/kubernetes/enhancements/issues/5004) —— DRA 设备发现/兼容性/节点准备/扩展资源桥接的一整套配套件同步推进。

**判断**:1.37 周期 DRA 从"能不能分一张卡"升级到"能不能把一张卡切片、按容量共享、跨多 Pod 工作负载统一声明、并对齐 NUMA"。我们若仍押注 device-plugin / 自研切分,窗口正在收窄。

### LWS / JobSet

- **LWS v0.9.0**(2026-06-17,[release notes](https://github.com/kubernetes-sigs/lws/releases/tag/v0.9.0)),本周最大的功能版本:
  - **DisaggregatedSet(KEP-766)首版落地** [proposal #767](https://github.com/kubernetes-sigs/lws/pull/767) + 实现 [#773](https://github.com/kubernetes-sigs/lws/pull/773)/[#836](https://github.com/kubernetes-sigs/lws/pull/836)/[#871(helm)](https://github.com/kubernetes-sigs/lws/pull/871):把推理的 **prefill/decode 解耦(disaggregated serving)** 做成按角色(role)编排的原生 set,带滚动更新。这是 LWS 从"对称多节点副本"走向"非对称分角色推理拓扑"的范式扩展,正好对上当前 vLLM/SGLang 的 P/D 分离潮流。
  - **RecreateGroupAfterStart 转正为 API** [#757](https://github.com/kubernetes-sigs/lws/pull/757):组内某 Pod 启动后失败 → 整组重建,故障恢复语义正式化。
  - 新增 **TLS 配置** [#783](https://github.com/kubernetes-sigs/lws/pull/783)、**status.observedGeneration** [#807](https://github.com/kubernetes-sigs/lws/pull/807)、多容器 Pod 的 **TPU 环境变量** [#754](https://github.com/kubernetes-sigs/lws/pull/754);修复 scale-up/zero-unavailable 滚动更新的 surge 处理 [#778](https://github.com/kubernetes-sigs/lws/pull/778);依赖升到 k8s 0.36、Go 1.26。
  - KEP 草案:[fail-fast restart budget + init-phase DNS](https://github.com/kubernetes-sigs/lws/pull/813)——大规模多节点推理的"快速失败重启预算"语义,值得跟。
- **JobSet**:本周仅一个依赖 bump([#1245](https://github.com/kubernetes-sigs/jobset/pull/1245)),无功能变化,最新仍 v0.12.0(2026-05)。

### scheduler-plugins(gang / NUMA)

- NUMA 拓扑(NRT)三连:[#978 overreserve 只计 sidecar 容器](https://github.com/kubernetes-sigs/scheduler-plugins/pull/978)、[#958 cache 修 assumedResource 漂移](https://github.com/kubernetes-sigs/scheduler-plugins/pull/958)、[#976 用配置的 rest.Config](https://github.com/kubernetes-sigs/scheduler-plugins/pull/976)——NUMA 感知调度的资源记账精度修复。

## KEP 动向

本周 enhancements 的 AI 相关合入集中在 **1.37 的 DRA 与"工作负载感知调度(WAS)"** 两条主线。DRA 已在上面 DRA 段详列,这里补 WAS / gang / 调度主线:

- [KEP-6012 CompositePodGroup API(初版文档)](https://github.com/kubernetes/enhancements/pull/6017) —— **原生 gang scheduling 的组合 PodGroup API**。若 SIG-Scheduling 把 gang 收进核心,长期看会蚕食 Volcano/Coscheduling 第三方插件的地盘,我们的 gang 选型要留意。
- [KEP-5732 Topology-aware workload scheduling → Beta(1.37)](https://github.com/kubernetes/enhancements/pull/6134) —— 拓扑感知的工作负载调度升 beta,和 Kueue TAS 形成上下游呼应。
- [KEP-5710 Workload-aware preemption → Beta(1.37)](https://github.com/kubernetes/enhancements/pull/6106) —— 工作负载级抢占(而非单 Pod 抢占),对训练作业的整组抢占语义更友好。
- [KEP-6089 WAS Controller Integration APIs](https://github.com/kubernetes/enhancements/pull/6092) + [KEP-5547 Integrate Workload APIs with Job controller](https://github.com/kubernetes/enhancements/pull/6153) —— WAS 控制器集成 API + 与 Job controller 打通,WAS 正在从设计走向落地。
- [KEP-6132 PreQueueingHint Extension Point](https://github.com/kubernetes/enhancements/pull/6133) —— 调度器事件处理新增 PreQueueingHint 扩展点,影响所有自定义调度插件的入队效率。
- 旁支:[KEP-4188 → Beta(1.37)](https://github.com/kubernetes/enhancements/pull/6126)、[KEP-5328 Node Declared Features → GA](https://github.com/kubernetes/enhancements/pull/6086)(节点声明特性 GA,设备/能力广告的底座)。

- [ ] **WAS / CompositePodGroup(KEP-6012)+ 拓扑感知调度升 beta(KEP-5732)**:判断 gang/拓扑调度往核心收编的节奏,评估是否减少对 Volcano/scheduler-plugins 的依赖。
