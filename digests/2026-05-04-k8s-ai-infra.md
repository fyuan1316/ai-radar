## 调度 & 资源分配

### Kueue
- [v0.18.0-rc.0 + v0.17.2 + v0.16.7 同日发布](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.18.0-rc.0) — 三线并发出版本
  - 启示:Kueue 维护节奏专业化,补丁线开始稳定回流
- 新能力(v0.18 主线):
  - [Concurrent Admission](https://github.com/kubernetes-sigs/kueue/pull/10610) — 单 ClusterQueue 内并行准入(此前是串行排队),大集群 throughput 改善
  - [MultiKueueRedoAdmissionOnEvictionInWorker → stable](https://github.com/kubernetes-sigs/kueue/pull/10695) — 跨集群驱逐后重投递可生产可用
  - [MultiKueueWaitForWorkloadAdmitted → stable](https://github.com/kubernetes-sigs/kueue/pull/10656) — manager 等 worker 实际 admit 后才返回
  - [SkipFinalizersForPodsSuspendedByParent → stable](https://github.com/kubernetes-sigs/kueue/pull/10645)
- 重大 bug 修复:
  - [TAS PodSet groups 空 topologyAssignment](https://github.com/kubernetes-sigs/kueue/pull/10783) — 拓扑亲和的核心场景
  - [ElasticJobs 扩容后 pod 仍 gated](https://github.com/kubernetes-sigs/kueue/pull/10272) — 弹性训练落地阻塞点
  - [HA 模式 follower 缓存陈旧](https://github.com/kubernetes-sigs/kueue/pull/10518) — 高可用部署的隐患
  - [强制删除 unreachable node 上 Failed/Succeeded pod](https://github.com/kubernetes-sigs/kueue/pull/10853) — JobSet foreground cascade 卡住的根因
  - 启示:v0.18 是 Kueue 真正"生产级 MultiKueue + TAS + Elastic"的版本,我们若做多集群 AI 平台,等 v0.18 GA 即可;在那之前,v0.17.2 是稳定线。重点检查 ElasticJobs / TAS 在我们部署的覆盖度
- [Kueue agent skills 实验](https://github.com/kubernetes-sigs/kueue/pull/10744) — `cmd/experimental/agent/skills` 新增 LLM agent skill 文件
  - 启示:连 Kueue 都开始内嵌"给 LLM agent 用的运维操作集";Kubernetes 生态对 agent-driven ops 的接纳速度比预期快

### DRA(Dynamic Resource Allocation)
本周 4 个值得跟的 KEP:
- [KEP-5963: Device Compatibility Groups](https://github.com/kubernetes/enhancements/pull/5964) — 描述"哪些设备可互替",影响 GPU + 加速器混编场景
- [KEP-5981: DRA Sharing Affinity for Conditional Fungibility](https://github.com/kubernetes/enhancements/pull/5987) — 在哪些条件下设备可被共享/可替代
- [KEP-5055: Device Taints and Tolerations → GA](https://github.com/kubernetes/enhancements/pull/6040) — Taint/Toleration 模型落到设备级,GPU 故障隔离的核心机制
- [KEP-5729: ResourceClaim Workload Support(1.36 实现对齐)](https://github.com/kubernetes/enhancements/pull/5976) — Pod/Job 直接绑 ResourceClaim,不走 ResourceClaimTemplate
  - 启示:DRA 在 1.36 之后就是"GPU 调度新事实标准";Compatibility Groups + Sharing Affinity 出现意味着"昇腾 vs NVIDIA 混跑"这类场景在 DRA 层面有原生支持的可能;Device Taints GA 是产品故障隔离的兜底机制,我们 NPU 故障处置应迁移到 DRA Taints

### LWS / JobSet
- [JobSet: Elastic JobSet(pod-level scaling)](https://github.com/kubernetes-sigs/jobset/pull/1179) — 大型 feature merge
  - 启示:训练作业终于能"加 pod 不重建 Job",这是云原生分布式训练长期痛点;LWS + JobSet + Kueue ElasticJobs 三件套完整后,弹性训练框架已闭合,我们做训练平台无需自造编排
- [LWS: 加 status.observedGeneration](https://github.com/kubernetes-sigs/lws/pull/807) — 标准 controller 字段补齐
- [LWS README 增加 DisaggregatedSet 描述](https://github.com/kubernetes-sigs/lws/pull/840) — DisaggregatedSet 是 LWS 0.9 计划的"prefill/decode 分离推理"子集
  - 启示:LWS 已经把"P/D 分离推理"列入产品 roadmap,这是 vLLM/llm-d/Dynamo 的同构需求;若我们做推理产品,LWS DisaggregatedSet 是要优先评估的原生路径
