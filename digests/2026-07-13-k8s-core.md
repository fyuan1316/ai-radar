# K8s 自身核心周报 2026-07-13

窗口:2026-07-06 -> 2026-07-13(7 天)

## 摘要(3 条以内)
- [Kubernetes v1.37.0-alpha.3](https://github.com/kubernetes/kubernetes/releases/tag/v1.37.0-alpha.3) 发布，v1.37 周期进入更密集 alpha 验证窗口；本周 KEP 信号集中在 assigned resources downward API、PodGroup preemption、storage capacity、tolerations/node affinity 表达能力。
- 自动扩缩容侧信息量大:Cluster Autoscaler 发布 1.33/1.34/1.35/1.36 patch/minor，Karpenter v1.14.0 发布，并继续推进 VPA prediction、pod deletion cost、NodeClaim optimization、外部健康信号阻断 disruption。
- Runtime、安全策略和网关侧都有可落地更新:containerd 多分支 patch、Gatekeeper v3.23.0、Kyverno v1.18.2、Gateway API TelemetryPolicy/GEP-1619/证书校验相关讨论值得跟进。

## K8s 核心版本节奏

- [Kubernetes v1.37.0-alpha.3](https://github.com/kubernetes/kubernetes/releases/tag/v1.37.0-alpha.3) — v1.37 alpha 版本继续推进。
  - 启示:v1.37 的 KEP 窗口已经打开。AI 平台底座要继续跟踪 assigned resources、PodGroup/preemption、topology、autoscaling、storage capacity 这些会影响调度和资源 API 的 KEP。
- [KEP-6122 DownwardAPIAssignedResources](https://github.com/kubernetes/enhancements/pull/6230) — 更新 disabled behavior 和 kubelet downgrade 建议。
  - 启示:workload 通过 Downward API 感知被分配资源的能力会影响 DRA/GPU/NPU 容器内发现路径。升级/降级行为必须明确，否则节点回退可能影响容器内 resource metadata。
- [KEP-5710 PodGroupPreemptionPolicy](https://github.com/kubernetes/enhancements/pull/6222) — 更新 feature gate。
  - 启示:PodGroup 级抢占会影响 gang scheduling、JobSet、LWS 和 Kueue。我们需要确认不同批/推理控制器对 PodGroup preemption policy 的兼容。

## 调度(kube-scheduler)

- [KEP-5721 Semantic Version Comparison Operators to Tolerations and NodeAffinity](https://github.com/kubernetes/enhancements/pull/5722) 与 [KEP-5500 CEL expressions support for Tolerations](https://github.com/kubernetes/enhancements/pull/5822) 本周仍有更新/关闭活动。
  - 启示:如果 toleration/node affinity 支持更丰富表达式，节点池约束会从简单 label match 走向版本/能力表达。AI 节点调度可用它表达 driver、CUDA/CANN、固件、kernel 版本，但也要防止策略过度复杂。
- 通用 kube-scheduler 本周未确认到单独 release note；AI 专用拓扑/DRA/Kueue 变化已归入 `k8s-ai-infra`。

## 节点 & kubelet

- [KEP-6122](https://github.com/kubernetes/enhancements/pull/6230) 对 kubelet downgrade 场景给出建议。
  - 启示:节点组件回退是企业升级常态。涉及 assigned resources、DRA、in-place resize 的功能必须设计降级路径和 feature gate 关闭后的容器行为。

## 自动扩缩容(VPA / CA / Karpenter)

- [Cluster Autoscaler 1.36.0](https://github.com/kubernetes/autoscaler/releases/tag/cluster-autoscaler-1.36.0)、[1.35.1](https://github.com/kubernetes/autoscaler/releases/tag/cluster-autoscaler-1.35.1)、[1.34.4](https://github.com/kubernetes/autoscaler/releases/tag/cluster-autoscaler-1.34.4)、[1.33.5](https://github.com/kubernetes/autoscaler/releases/tag/cluster-autoscaler-1.33.5) 本周发布。
  - 启示:CA 与 K8s minor 的兼容节奏仍然紧。我们托管集群或 AI 节点池升级时，要把 CA 版本与控制面版本绑定验证，尤其是 Pending Pod 与队列系统的交互。
- [Autoscaler PR #9962](https://github.com/kubernetes/autoscaler/pull/9962) — AEP:Per-VPA Observation Window。
  - 启示:VPA 建议窗口如果可按对象配置，会更适合训练、推理、Notebook 这类负载差异大的场景。我们可以把“短作业/长服务/交互式 notebook”的 VPA profile 产品化。
- [Autoscaler PR #8516](https://github.com/kubernetes/autoscaler/pull/8516) — AEP:VPA object level 自定义 request-to-limit ratio。
  - 启示:AI workload 经常 request/limit 关系不等于通用服务。VPA 如果支持对象级 ratio，可以减少手工调参，但 GPU/NPU 资源仍需独立策略。
- [Autoscaler PR #9786](https://github.com/kubernetes/autoscaler/pull/9786) — RFC:integrate extender-managed resources。
  - 启示:扩缩容对“外部管理资源”的理解会影响 GPU/NPU、自定义拓扑资源和 DRA。我们应关注 CA/VPA 如何看待非 CPU/Mem 资源。
- [Karpenter v1.14.0](https://github.com/kubernetes-sigs/karpenter/releases/tag/v1.14.0) 发布，同时多条 patch release 更新。
  - 启示:Karpenter 仍是节点自动扩缩容的高频创新源。AI 节点池如果采用 Karpenter，需要关注 NodeClaim、disruption、consolidation 与 GPU workload drain/checkpoint 的关系。
- [Karpenter PR #3134](https://github.com/kubernetes-sigs/karpenter/pull/3134) — 增加 VPA prediction controller and store。
  - 启示:节点扩缩容正在尝试消费 workload 资源预测。AI 平台可以把历史利用率、队列需求和 VPA 建议结合，提前准备 GPU/NPU capacity。
- [Karpenter PR #2894](https://github.com/kubernetes-sigs/karpenter/pull/2894) 与 [#2935](https://github.com/kubernetes-sigs/karpenter/pull/2935) — pod deletion cost management controller / RFC。
  - 启示:缩容时尊重 Pod 删除成本对训练和推理很关键。我们应把 checkpoint、模型加载耗时、KV cache warmup 作为 deletion cost 输入。
- [Karpenter PR #2901](https://github.com/kubernetes-sigs/karpenter/pull/2901) — 外部健康信号阻断 NodePool disruption/consolidation 的 RFC。
  - 启示:AI 节点是否可中断取决于作业、checkpoint、硬件健康、网络拓扑，不只是 Karpenter 内部状态。平台应能注入外部健康/业务信号。

## Runtime(containerd / CRI-O / runc)

- [containerd 2.3.3](https://github.com/containerd/containerd/releases/tag/v2.3.3)、[2.2.6](https://github.com/containerd/containerd/releases/tag/v2.2.6)、[2.0.11](https://github.com/containerd/containerd/releases/tag/v2.0.11)、[1.7.34](https://github.com/containerd/containerd/releases/tag/v1.7.34) 本周发布。
  - 启示:containerd 多分支 patch 说明 runtime 安全/稳定更新要覆盖不同客户基线。GPU/NPU runtime、CDI、image pull、snapshotter 回归应按 containerd 主流分支做矩阵。
- 本窗口未确认到 CRI-O / runc 与 AI 平台直接相关的新 release。

## API Machinery & Operator 生态

- [controller-runtime PR #3473](https://github.com/kubernetes-sigs/controller-runtime/pull/3473) / [#3472](https://github.com/kubernetes-sigs/controller-runtime/pull/3472) — read-your-own-write client design / POC。
  - 启示:operator reconcile 经常受 cache 延迟影响。AI 平台 operator 在创建复杂 DAG 资源后读状态，如果能明确 RYOW 语义，可减少偶发竞态。
- [controller-runtime PR #3540](https://github.com/kubernetes-sigs/controller-runtime/pull/3540) 与 release 分支 PR [#3548](https://github.com/kubernetes-sigs/controller-runtime/pull/3548)、[#3549](https://github.com/kubernetes-sigs/controller-runtime/pull/3549) — 修复 PriorityQueue shutdown deadlock。
  - 启示:operator shutdown、leader election、queue drain 的边界会影响升级稳定性。我们的 operator 升级测试要覆盖 graceful shutdown 与 leader 切换。
- [controller-runtime PR #3481](https://github.com/kubernetes-sigs/controller-runtime/pull/3481) — webhook server port 设为 -1 可禁用 webhook server。
  - 启示:组件可选 webhook 对离线、权限受限和最小安装场景有用。我们应支持按 profile 禁用不必要 webhook。

## 网络 & 存储(大盘)

- [Gateway API PR #4872](https://github.com/kubernetes-sigs/gateway-api/pull/4872) — GEP-4768 TelemetryPolicy API proposal。
  - 启示:MaaS/LLM endpoint 的观测策略可能进入 Gateway API。我们应关注是否能用 TelemetryPolicy 统一请求日志、指标、trace，而不是每个模型服务单独配置。
- [Gateway API PR #4873](https://github.com/kubernetes-sigs/gateway-api/pull/4873) 与 [#5046](https://github.com/kubernetes-sigs/gateway-api/pull/5046) — GEP-1619 SessionPersistence 调整 cookie path / union pattern。
  - 启示:LLM 多轮会话、sticky routing、KV cache affinity 都依赖会话保持。Gateway API 的 session persistence 语义会影响推理路由实现。
- [Gateway API PR #4079](https://github.com/kubernetes-sigs/gateway-api/pull/4079) — client certificate validation 的 certificate pinning GEP。
  - 启示:企业 MaaS 接入经常要求 mTLS 和证书约束。Gateway 层证书 pinning 能减少自研网关安全扩展。
- [KEP-5381 CSI ControllerModifyVolume topology integration](https://github.com/kubernetes/enhancements/pull/6231) 与 [KEP-5925 enhanced storage capacity tracking](https://github.com/kubernetes/enhancements/pull/5926) 本周活跃。
  - 启示:模型权重、checkpoint、KV cache offload 都依赖存储容量和拓扑。CSI 容量跟踪与拓扑修改能力会影响模型数据放置。

## 策略 & 安全

- [Gatekeeper v3.23.0](https://github.com/open-policy-agent/gatekeeper/releases/tag/v3.23.0) 发布。
  - 启示:Gatekeeper 仍是企业策略基线。AI 平台要把模型服务、GPU/NPU 节点、Notebook Secret 引用、MaaS endpoint 的约束模板纳入 Gatekeeper 兼容验证。
- [Kyverno v1.18.2](https://github.com/kyverno/kyverno/releases/tag/v1.18.2) 发布；本周 PR 侧还包括 cache bounds、readiness checker HTTPS 证书默认校验、image verification credential handling refactor。
  - 启示:Kyverno 在镜像验证、报告、webhook readiness 上继续强化。我们的离线制品、模型镜像、runtime sidecar 应能被 Kyverno/Gatekeeper 双栈策略覆盖。

## KEP 动向(非 AI)

- [KEP-6228 JSON Pointer extraction for Secret and ConfigMap data](https://github.com/kubernetes/enhancements/pull/6229)。
  - 启示:如果 Secret/ConfigMap 数据提取更细粒度，平台可以更安全地把配置片段注入工作负载，但也要重新评估权限和审计边界。
- [KEP-4785 RSM beta](https://github.com/kubernetes/enhancements/pull/5766)。
  - 启示:资源/调度模型的 beta 晋级要关注与队列、PodGroup 和拓扑资源的交互。
- [ReferenceGrant KEP broadened to Referential Authorization](https://github.com/kubernetes/enhancements/pull/4387)。
  - 启示:跨 namespace 引用授权是 Gateway、Secret、模型资源、共享缓存的关键基础。AI 平台需要统一引用授权模型。
