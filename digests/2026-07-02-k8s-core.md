## K8s 核心版本节奏

- 主仓本周无新 tag,仍停在 [v1.37.0-alpha.2](https://github.com/kubernetes/kubernetes/releases/tag/v1.37.0-alpha.2)(v1.37 周期)与 v1.36.2 稳定线。窗口内的信号主要来自 enhancements 仓的 KEP 状态推进,而非二进制 release。
  - 启示:v1.37 仍在 alpha 收集特性阶段,是把 KEP watchlist 落成"进哪个 minor 的 alpha/beta/GA"的最佳时机。下面 sig-node / sig-scheduling / sig-autoscaling 的几条都要标注目标版本,别等 beta 才发现产品 API 受影响。

## 调度(kube-scheduler)

- [KEP-5980 TopologySpreadConstraint skew excludes fully-tainted domains](https://github.com/kubernetes/enhancements/pull/6210)(sig-scheduling,open):当 `nodeTaintsPolicy=Honor` 且某拓扑域内所有节点都被 taint 时,该域仍计入 `minDomains` 但从 skew 计算中排除,从而在整可用区故障时不阻塞 Pod 调度。
  - 启示:对多可用区部署的 AI 平台是实用修复。整区节点被 taint(升级、隔离、故障)时,现状可能因 skew 判定把新 Pod 全部卡 Pending。若我们对推理/训练负载启用 topologySpread + Honor,应跟这条到落地版本,并在测试里覆盖"整区不可用"场景。
- DRA 侧 [KEP-5007 Device Binding Conditions 计划 1.38 GA](https://github.com/kubernetes/enhancements/pull/6005) 有更新,但 DRA 设备层归 `k8s-ai-infra`,此处仅备注版本节奏,不展开。

## 节点 & kubelet

- [KEP-6063 per-pod PID limit 最小值 1024 → 128](https://github.com/kubernetes/enhancements/pull/6217)(sig-node,新建于 06-30):基于 OpenShift 集群实测(7 容器 Pod + exec 探针 + PreStop 后台进程也只用到很少 PID),把 `spec.resources.limits.pid` 允许的最小值下调到 128。
  - 启示:高密度节点上,更细粒度的 PID 限制有助于抑制 fork bomb / 进程泄漏对邻居的影响。我们做多租户节点池时可以把 per-pod PID 限制纳入 profile,但要注意含大量 exec 探针 / sidecar 的推理 Pod 的真实 PID 峰值,别把下限设得过激导致探针 fork 失败。
- [KEP-4438 sidecar 终止语义对齐实现](https://github.com/kubernetes/enhancements/pull/6218)(sig-node,更新于 07-01):重启的 sidecar(restartable init container)现在在终止时收到有序 `SIGTERM` 而非 sandbox 拆除时的 `SIGKILL`;该重启不体现在 pod status,改为暴露 `kubelet_sidecar_restarts_during_termination_total` 指标;`preStop` 在 t=0 只跑一次。
  - 启示:直接影响所有用 native sidecar(如 mesh sidecar、日志/指标 agent、模型代理)的负载的优雅退出。我们如果依赖 sidecar 在 Pod 终止阶段 flush 数据,要确认它能正确响应 SIGTERM,并把这个新指标接进可观测,判断终止期 sidecar 是否被异常重启。
- [KEP-2570 memoryThrottlingFactor 默认值改 nil(v1.37)](https://github.com/kubernetes/enhancements/pull/6216)(sig-node,已合并 06-25):调整 cgroup v2 memory.high 节流因子在 v1.37 的默认行为。
  - 启示:cgroup v2 memory throttling 默认语义变化会影响内存敏感型工作负载(大模型加载、KV cache)。升级到 1.37 前应在压测里对比 memory.high 节流对 OOM/性能的影响,必要时显式设置该因子而非依赖默认。

## 自动扩缩容(VPA / CA / Karpenter)

- VPA in-place resize 线继续打磨:autoscaler mono-repo 本周有 `Add in-place infeasible cached metric`、`vpa: remove redundant in-place-skip-disruption-budget log` 等提交([kubernetes/autoscaler commits](https://github.com/kubernetes/autoscaler/commits/master))。
  - 启示:VPA + in-place pod resize(KEP-1287 线)是 VPA 从"准 alpha"走向生产可用的关键。新增"in-place infeasible"缓存指标说明社区在观测"原地调整失败需回退重建"的比例——这正是我们评估 VPA 能否在 AI 推理服务上开启的核心风险点,值得持续跟。
- Cluster Autoscaler:本周主要是 ClusterStateRegistry 判定 upcoming Nodes / 已删除 Node 的逻辑统一化 + 测试补齐,以及 `Introduce ObservedGeneration to CapacityQuota`、GCE `RefreshMigInfo`、Azure `report deleting instances as gone` 等 provider 侧修复([同上](https://github.com/kubernetes/autoscaler/commits/master))。
  - 启示:CA 对"节点即将上线 / 已删除"的状态一致性直接影响扩容抖动。多云环境(尤其 GPU 节点池扩容慢)下,这类一致性修复能减少误判重复扩容,升级 CA 时可留意。
- HPA:[KEP-5679 external-metric-fallback](https://github.com/kubernetes/enhancements/pull/6213)(sig-autoscaling)新增 fallback 策略字段 `Static / UseMax / UseMin`,外部指标源不可用时按策略回退而非直接停止扩缩。
  - 启示:基于外部指标(如 Prometheus adapter、队列长度、token 吞吐)做 HPA 的推理服务,最怕指标管道抖动导致 HPA 冻结。这条 KEP 落地后可配置"降级到静态副本 / 取历史最大"作兜底——是我们 MaaS 弹性策略应直接采用的能力。
- Karpenter:本窗口无新 release。

## API Machinery & Operator 生态

- [KEP-5933 x-kubernetes-sensitive-data marker for CRD fields](https://github.com/kubernetes/enhancements/pull/5937)(sig-api-machinery,open):新增 `x-kubernetes-sensitive-data` vendor extension,标记后对未授权调用者在 get/list/watch 时剥离该字段,并在审计日志里无条件脱敏。
  - 启示:对我们自研 CRD 是很实用的合规原语。模型服务 CRD 里的 token、连接串、许可证 key 之类敏感字段,未来可用一个 marker 同时拿到"RBAC 之外的字段级隐藏 + 审计脱敏",省掉自己写 webhook。企业级安全合规路线图可把它列入候选。
- Operator SDK [v1.42.3](https://github.com/operator-framework/operator-sdk/releases/tag/v1.42.3)(06-26)已在上周 digest 覆盖,主要是依赖升级,本周不重复。
- controller-runtime / kubebuilder 本窗口无新 release。

## 网络 & 存储(大盘)

- [Gateway API v1.6.0](https://github.com/kubernetes-sigs/gateway-api/releases/tag/v1.6.0)(GA,06-29)相对 v1.5.1 的重点:
  - UDPRoute、TCPRoute **晋级 GA**,推荐用 `v1` 版本,v1alpha2 弃用待移除。
  - BackendTLSPolicy 现可与其他路由类型组合使用;CA(Certificate Authority)引用上限 8 → 16;网关基础设施对象注解上限提到 16。
  - HTTPRoute retry 校验收紧:`retry.codes` 必须唯一、`retry.attempts` 必须 ≥ 1;TLSRoute 放开到最多 1024 hostname/rule(生产启用前需验证 apiserver/etcd/控制器行为)。
  - 文档站从 MkDocs 迁到 Docsy,新增配置向导 https://gateway-api.sigs.k8s.io/wizard 。
  - 启示:TCP/UDP L4 路由 GA,意味着 Gateway API 不再只覆盖 HTTP/gRPC——对需要暴露非 HTTP 协议端点(如某些推理/向量库/RDMA 控制面)的场景,可以统一到 Gateway API 而非另配 Service LB。我们的网关/安装器应:1) 校验 HTTPRoute retry 新约束避免升级后校验失败;2) 评估把 L4 endpoint 收敛到 UDPRoute/TCPRoute v1。
- [Calico v3.32.1](https://github.com/projectcalico/calico/releases/tag/v3.32.1)(06-26)已在上周 digest 覆盖,本周不重复。
- Cilium 本窗口无新 release;CSI(external-provisioner/snapshotter/attacher)本窗口无新 release。
- 存储 KEP 在推进但均未合并:[KEP-6058 CSI volume reconstruction fallback](https://github.com/kubernetes/enhancements/pull/6059)、[KEP-5925 enhanced storage capacity tracking](https://github.com/kubernetes/enhancements/pull/5926)(均 sig-storage,open)。
  - 启示:capacity tracking 增强与调度对 PV 容量的感知相关,若我们做有状态 AI 负载(数据集缓存、checkpoint PVC)的容量感知调度,可提前跟这条。
