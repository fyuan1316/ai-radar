# K8s AI 基础设施周报 2026-07-09

窗口:2026-07-02 → 2026-07-09。本窗口内 AI-specific 组件**无新 release**,信号集中在合入的 PR 与 KEP 动向;DRA 侧有一项在 1.37 落地的 GA。

## 摘要(3 条以内)

1. **DRA 是本周最强信号:Device Taints & Tolerations 在 1.37 GA(07-06 合入,`resource.k8s.io/v1`)**。配合本 1.37 周期已 GA 的 **DRA Extended Resource**(pod 用标准 `resources.requests: nvidia.com/gpu` 请求,kube-scheduler 依据 DeviceClass 的 `extendedResourceName` 自动创建 ResourceClaim),device-plugin → DRA 的迁移路径在 1.37 基本闭环:既能像老模型那样按扩展资源名申请,又能用 DRA 的污点/容忍做设备级排空与隔离。这是我们加速器方案选型必须跟到底的主线。https://github.com/kubernetes/kubernetes/pull/138676
2. **Kueue 的 DRA 配额(KEP-2941)推进 beta/GA 毕业标准,覆盖 Extended Resources 与 Partitionable Devices 两条路径**(07-08 合入)。PD = 对可动态分片设备(如 NVIDIA MIG)按 counter 计量配额,而非按整卡计数——多租户 GPU/MIG 配额治理正在 Kueue 里原生化,直接对标我们自研的租户 GPU 护栏。同期 Workload 的 PodSet 上限从 10 提到 18,支撑更多角色/模板的大作业。https://github.com/kubernetes-sigs/kueue/pull/12424
3. **LWS 给 DisaggregatedSet 加 `spec.slices` 字段**(07-08 合入):把 prefill/decode 分离式推理的整套角色拓扑一键复制成 N 份独立副本、各自独立滚动更新。这是分离式(P/D disaggregation)推理规模化的基座 API,OAI 推理平面方向要重点盯。https://github.com/kubernetes-sigs/lws/pull/873

## 调度 & 资源分配

### Kueue

活跃度最高(窗口内 121 个合入 PR,多为测试/cherry-pick,择要):

- **KEP-2941(DRA)beta/GA 毕业标准更新,ER + PD 两条路径**(#12424,07-08):ER = Extended Resources(标准 `resources.requests` 请求 DRA 设备);PD = Partitionable Devices,对 MIG 这类可分片设备按 counter 做配额。这是 Kueue 把 GPU/MIG 配额治理原生化的关键一步。https://github.com/kubernetes-sigs/kueue/pull/12424
- **Workload PodSet 上限 10 → 18**(#12819):`MaxPodSets` 常量与 v1beta1/v1beta2 五个 podSet 数组的 MaxItems 校验同步抬升,支撑角色/模板更多的大作业(如多角色分离式推理、复杂 JobSet)。https://github.com/kubernetes-sigs/kueue/pull/12819
- **`WaitingForReplacementPods=True` 时不 admit 工作负载**(#12768,已 cherry-pick 到 0.17/0.18):修复弹性/容错作业在等待替换 Pod 期间被错误准入的问题,关乎故障恢复正确性。https://github.com/kubernetes-sigs/kueue/pull/12768
- **安全:kueueviz 修复 CSWSH(跨站 WebSocket 劫持)**,移除过于宽松的 `CheckOrigin`,并补 http server 超时,全部 cherry-pick 到 0.17/0.18(#12734/#12590 等)。若在生产暴露 kueueviz 面板需升级。https://github.com/kubernetes-sigs/kueue/pull/12734
- KEP-8190(TLS security profiles)加椭圆曲线配置(#11832)、KEP-9270 MultiKueue 增量派发 step size 文档澄清(#12853)——安全/多集群运维向,信号中等。https://github.com/kubernetes-sigs/kueue/pull/11832

### DRA

DRA 是本任务重中之重(将取代 device-plugin,影响所有加速器方案):

- **Device Taints & Tolerations → GA(1.37,`resource.k8s.io/v1`)**,#138676 于 07-06 合入。可对 ResourceSlice/设备打污点、由 ResourceClaim 声明容忍,实现设备级排空、隔离坏卡、维护窗口。https://github.com/kubernetes/kubernetes/pull/138676
- **DRA Extended Resource → GA(1.37 周期,#138488,06-03 合入,随本周 v1.37.0-alpha.3 首次进入发布线)**:pod 用标准扩展资源名申请、scheduler 自动建 ResourceClaim,是 device-plugin 用户平滑迁移 DRA 的桥。https://github.com/kubernetes/kubernetes/pull/138488
- 本周 DRA 稳定性修复:PodGroup 内共享同一 ResourceClaim 的 Pod 卡调度(#140269,07-07)、kubelet 删 Pod 时误 unprepare 仍被他 Pod 使用的资源(#140212,07-07)。gang + DRA 组合路径正在被打磨。https://github.com/kubernetes/kubernetes/pull/140269
- **KEP-5007 DRA Device Binding Conditions 的 GA 毕业 PR(#6005)仍在途**,里程碑在 1.37/1.38 间待定(PR 标题写 1.38、正文写 1.37,尚未定稿),未合入。该特性让调度在设备"绑定条件"(如网络/拓扑就绪)满足后才最终绑定,对多节点训练的设备编排关键,持续盯。https://github.com/kubernetes/enhancements/pull/6005

### LWS / JobSet

- **LWS DisaggregatedSet 加 `spec.slices`**(#873,07-08):整套分离拓扑复制 N 份独立副本、独立滚动——P/D 分离式推理规模化基座(详见摘要 3)。同期补 DisaggregatedSet API 参考文档(#878/#811)。https://github.com/kubernetes-sigs/lws/pull/873
- LWS 健壮性:webhook 拒绝 replicas 越界 `[0, 1000000]`(#913)、修复 Helm webhook 的 cert-manager CA 注入(#909)、修 `enablePrometheus` 的 ServiceMonitor(#903)。https://github.com/kubernetes-sigs/lws/pull/913
- **JobSet**(无窗口内 release,最新 v0.12.0 为 05-08):修复 reconcile 时 nil network 导致的 panic(#1266)、generateName 派生子资源名的校验(#1267)——都是大规模训练作业编排的稳定性补丁。https://github.com/kubernetes-sigs/jobset/pull/1266
- **scheduler-plugins**(无窗口内 release,最新 v0.34.7 为 04-20):NodeTopologyMatch 改进独占资源探测(#984);新增 KEP-942 NodeMetadata scoring plugin(#974,见下)。https://github.com/kubernetes-sigs/scheduler-plugins/pull/984

## KEP 动向

> 本任务口径:只收 AI/加速器/调度专用 KEP。通用 KEP(sig-node/storage/auth 等)已由 `k8s-core` 覆盖,不重复。

- **KEP-5007 DRA Device Binding Conditions → GA(在途,#6005)**:GA 毕业 PR 开着,里程碑 1.37/1.38 未定稿。DRA GA 化的下一块拼图,决定多节点训练设备编排的成熟度。https://github.com/kubernetes/enhancements/pull/6005
- **KEP-6089 Workload Aware Scheduling(WVA)Controller APIs**:初稿 PR #6092 已于 06-16 合入,本窗口有后续讨论。这是 gang / 工作负载感知调度的上游 API 层(与 1.37 主线已进的 PodGroup API 同源),将影响所有 gang scheduling 生态(scheduler-plugins Coscheduling、Kueue、Volcano)的收敛方向,是决定我们 6 个月后调度选型的关键线。https://github.com/kubernetes/enhancements/pull/6092
- **scheduler-plugins KEP-942 NodeMetadata scoring plugin(#974,07-03 合入)**:按节点 label/annotation 的数值或时间戳打分(如按节点新旧、GPU 代次择优),给自定义节点优选提供通用插件。https://github.com/kubernetes-sigs/scheduler-plugins/pull/974
- **KEP-5951 调度器优先级队列 MoveAllToActiveOrBackoffQueue 批处理(#5952,在途)**:降低大量 Pod 同时重入队列时的调度器抖动,大规模 AI 集群(万级 Pod 排队)吞吐相关。https://github.com/kubernetes/enhancements/pull/5952
- 交界提示:**KEP-5726 TopologyManager CPU-attached NUMA filter**(#5992)对延迟敏感 / 高性能 AI 负载的 NUMA 亲和有影响,但归 `k8s-core`(sig-node),此处仅提示存在。https://github.com/kubernetes/enhancements/pull/5992
