# K8s 自身核心周报 2026-04-21

窗口:2026-04-14 → 2026-04-21
扫描范围:kubernetes/kubernetes + enhancements + autoscaler + karpenter + cluster-api + containerd + cri-o + controller-runtime + kubebuilder + gateway-api + cilium + gatekeeper + kyverno 共 13 个仓库,按"通用 K8s 能力"口径筛选(AI-specific 组件见 k8s-ai-infra)。

## 摘要(3 条)

1. **K8s v1.36.0-rc.1 发布(4/15),离 GA 只剩最后一步**;同日 v1.35.4 / v1.34.7 / v1.33.11 三个稳定分支出 patch。v1.36 自身的主要特性在 rc.0/beta(更早),rc.1 只做依赖更新类收尾。下一步就是 v1.36.0 GA,我们的客户侧 release 基线要开始准备从 1.35 跳 1.36([release](https://github.com/kubernetes/kubernetes/releases/tag/v1.36.0-rc.1))。
2. **containerd 全家桶统一修 spdystream CVE-2026-35469**(1.7.31 / 2.0.8 / 2.1.7 / 2.2.3 四条支持分支于 4/14 同时出 patch);同时 containerd 2.3.0-beta.2(4/17)宣告 2.3 周期正式切换到"与 K8s release cadence 对齐"的新节奏——containerd 开始每个 K8s minor 跟一个对应 containerd 版本([v2.2.3 release notes](https://github.com/containerd/containerd/releases/tag/v2.2.3))。
3. **KEP 动向两条主线**:**(a) VPA 生产化前序推进** — KEP-5836(Scheduler Preemption for In-Place Pod Resize,sig-scheduling,alpha)正在主仓更新,这是 In-Place Pod Resize(KEP-1287)GA 之后 VPA 真正可用的关键补完件;**(b) sig-apps 补齐扩缩容语义** — KEP-5982(ReplicaSet Consolidation-Aware Scale-In Strategy)、KEP-5682(PDBs for multi-pod replicas)本周同步推进,弥补"ReplicaSet/HPA 与 PDB 协调不够细"的长期缺口。

## 调度(kube-scheduler)

**本周主仓无新 release,全部信号在 KEP 层**(按 sig/scheduling label 过滤,updated in window):

- **[KEP-5836: Scheduler Preemption for In-Place Pod Resize (alpha)](https://github.com/kubernetes/enhancements/pull/5932)** — **最值得跟**。这是 In-Place Pod Resize(KEP-1287)GA 之后的关键补完件:当 in-place resize 抢不到资源时,由 scheduler 参与驱逐低优先级 pod 做让位。现状 VPA+in-place 在高负载节点就会降级回重启模式,这个 KEP 正面解决。
- **[KEP-6012: CompositePodGroup API (initial docs)](https://github.com/kubernetes/enhancements/pull/6017)** — sig-scheduling 的新 API,用于表达"一组相互关联的 pod 作为一个调度单位"(比 PodGroup 更结构化),对 AI workload 的 gang scheduling 也有溢出影响,但当前 KEP 本身是 sig-scheduling 视角的通用设计。
- **[KEP-5729: ResourceClaim for Workloads](https://github.com/kubernetes/enhancements/pull/5976)** — 基于 1.36 实现的更新。这个跟 DRA 强相关(k8s-ai-infra 同时写了),本任务视角是它把 ResourceClaim 抽到 Workload 层、对所有用 DRA 资源的控制器都有 API 影响。
- **[KEP-5517: 调度优化 alpha 实现更新](https://github.com/kubernetes/enhancements/pull/6009)**(4/17 MERGED) — sig-scheduling。
- **[KEP-5004: DRA Extended Resources promotion to GA](https://github.com/kubernetes/enhancements/pull/6023)** — 主线由 k8s-ai-infra 追,此处只备注:GA 后所有 Extended Resource 消费者(非 AI 也有)都要跟进 API 迁移。

**对我们产品的启示**:KEP-5836 是"VPA 真生产可用"最后一块拼图,跟进时要同步评估我们推理/训练平台里对 pod resize 的依赖。CompositePodGroup 如果走向 GA,gang scheduling 这一块上游会有一等公民 API,自研 PodGroup 需要有迁移计划。

## 节点 & kubelet

- **[KEP-5620: Node resizing via balloons](https://github.com/kubernetes/enhancements/pull/5624)** — sig-node(4/17 更新)。"balloons" 是一个借鉴虚拟化的比喻:在节点上预留可挤压的占位资源,当需要给真实负载腾容量时"放气"。这是 **节点级 vertical scale** 的新思路,不同于传统 cluster-autoscaler 的"加节点",而是让单节点本身弹性更强——对 spot 回收、GPU 共享、混部都有价值。
- **[KEP-5996: Add KEP initial README](https://github.com/kubernetes/enhancements/pull/5998)** — sig-node 新 KEP(4/20 更新),标题为"initial README",具体内容待跟进。
- **[KEP-5963: DRA Device Compatibility Groups](https://github.com/kubernetes/enhancements/pull/5964)** — 虽在 sig-node,内容是 DRA 延伸,主要由 k8s-ai-infra 追。

**对我们产品的启示**:KEP-5620 如果走通,是我们"客户混部 / Spot / GPU 分时"叙事的重要弹药;值得在下一个季度 roadmap 调研里加一个"Node balloons 评估"条目。

## 自动扩缩容(VPA / Cluster Autoscaler / Karpenter)

**本周 `kubernetes/autoscaler` 和 `kubernetes-sigs/karpenter` 均无新 release**,但 KEP 层看:

- 见上文 **KEP-5836(VPA 相关的 scheduler preemption)**
- **[KEP-5982: ReplicaSet Consolidation-Aware Scale-In Strategy](https://github.com/kubernetes/enhancements/pull/6019)**(sig-apps,4/16)— 让 ReplicaSet 在缩容时优先挑"节点碎片化更高"的 pod 删,配合 CA/Karpenter 的节点合并。之前是按 pod age / priority 删,跟节点利用率无关。
- **[KEP-5682: PDBs for multi-pod replicas](https://github.com/kubernetes/enhancements/pull/5671)**(sig-apps,4/17)— PDB 长期只基于单 replica,对 LWS / JobSet / StatefulSet 这种"一个逻辑副本 = 多个 pod"的场景不准。这个 KEP 扩 PDB 语义。

**对我们产品的启示**:VPA 生态(in-place resize → preemption → 真能上生产)这条线要在下半年单独列一个跟踪项;autoscaler mono-repo 本身没动不代表 VPA/CA 没事,KEP 是真信号。

## Runtime(containerd / CRI-O / runc)

**containerd 本周压倒性主角**,一次 release 了 6 个 tag(4/14~4/17):

- **Security**: 全部 4 个支持分支(1.7 / 2.0 / 2.1 / 2.2)**同步修 CVE-2026-35469(spdystream)**。`moby/spdystream` 是 kubelet exec/portforward 路径的依赖,影响面不小。所有线上 containerd 都应在一周内升级。
- **v2.2.3**(4/14,https://github.com/containerd/containerd/releases/tag/v2.2.3):CRI 侧保留 privileged 容器的 cgroup mount options(之前有丢失)。
- **v2.1.7**(4/14,https://github.com/containerd/containerd/releases/tag/v2.1.7):CRI 侧保留宿主 cgroup mount options。
- **v2.0.8**(4/14,https://github.com/containerd/containerd/releases/tag/v2.0.8):CRI 在 gRPC 返回前 sanitize 错误。
- **v1.7.31**(4/14,https://github.com/containerd/containerd/releases/tag/v1.7.31):CNI DEL 永不执行的场景修复。
- **v2.3.0-beta.2**(4/17,https://github.com/containerd/containerd/releases/tag/v2.3.0-beta.2):**2.3 周期正式切换到"与 K8s release cadence 对齐"**——每个 K8s minor 跟一个 containerd minor。transfer service 新增容器文件系统 copy 类型,shim bootstrap protocol 引入。对运维而言:containerd 升级节奏以后会跟 K8s minor 同步,不再孤岛演进。
- **api/v1.11.0-beta.2**(4/15,https://github.com/containerd/containerd/releases/tag/api/v1.11.0-beta.2):API 分离的版本,与 2.3 对齐。

**CRI-O**、**runc**:本周无新 release。

**对我们产品的启示**:立即跟 CVE-2026-35469,把它列入我们发行版的强制升级项;同时开始评估"containerd 与 K8s 同步节奏"对我们发行版打包流程的影响(意味着每个 K8s minor 都要同步验证对应 containerd minor,而不是随便挑一个长稳 1.7 用下去)。

## API Machinery & Operator 生态

**本周 `controller-runtime` / `kubebuilder` / `operator-sdk` 均无新 release**,API machinery 侧信号在 KEP:

- **[KEP-5958: Client Opt-out for managedFields in API Response](https://github.com/kubernetes/enhancements/pull/6015)**(sig-api-machinery,4/21)— 允许 client 请求时声明"不要 managedFields",直接降低 apiserver 响应体积。对 kubectl / controller-runtime 客户端都有性能收益,尤其是大规模 informer。
- **[KEP-4192: Set SVM to stable in 1.37](https://github.com/kubernetes/enhancements/pull/6020)**(sig-api-machinery,4/20)— StorageVersionMigrator 升 stable,意味着 CRD / 内置资源在 API 版本迁移时的自动迁移工具到达生产级,受益的主要是大集群升级场景。
- **[KEP-5673: Graceful Deletion support for Custom Resources](https://github.com/kubernetes/enhancements/pull/5674)**(sig-api-machinery,4/15 CLOSED 不 merge)— 探索 CRD 级别的 graceful deletion,本周关掉,方案需要重议。

**对我们产品的启示**:KEP-5958 对我们控制面性能有直接收益,升级到支持该特性的 client-go 版本后,informer / list 响应可瘦身 20-40%;KEP-4192 到 stable 意味着我们 operator/CR schema 迁移工具可以从"自研"改为"用上游 SVM"。
