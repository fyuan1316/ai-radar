## 对我们产品的启示

### DRA 选型(首要)

1. **DRA 在 1.36 已经是"默认开启 + 大部分核心特性 beta/GA"的状态**:PrioritizedList/AdminAccess 升 GA、DeviceTaints/PartitionableDevices/BindingConditions/ExtendedResource 全部升 beta。1.37 计划 ExtendedResource 升 GA —— 意味着下一个版本起,**基于 DRA 的 GPU 管理替代 device-plugin 的技术可行性在上游已经走通**。我们 ACP 如果还在 1.32/1.33 水位,需要尽早规划 DRA 的进入窗口(至少 1.34,最好 1.35/1.36),否则 NVIDIA/AMD 新形态(MIG 分区、混合分区、动态 partitioning)会越来越难靠 device-plugin 补齐。
2. **DRA Device Compatibility Groups(KEP-5963)是一个新的上游抽象**,正好覆盖我们内部评估里"混合启用 MIG + MPS 会冲突"的实际问题。建议跟踪后续 PR,看是否能直接用上游 API 描述,而不是自己在 webhook 里做排斥检查。
3. **NVIDIA GPU Operator v26.3.x 的 NRI Plugin 路径** 解决了"运行时注入强依赖 nvidia runtime class"的问题,对 k3s、Rancher、未来我们做边缘/轻量化 ACP 有直接价值。NRI Plugin 也是 DRA 路径落地到运行时的自然结合点,值得安排一次对比测试。
4. **Scheduler Plugins v0.34.7 给 scheduler SA 开放了 `resource.k8s.io`**,意味着 coscheduling 与 DRA 可以组合使用(gang scheduling + DRA)。我们如果走 coscheduling + DRA 路线,这个 RBAC 修复是必要依赖。

### LWS / DisaggregatedSet(产品形态级影响)

1. **DisaggregatedSet 官方定义了"分离式推理"的 K8s 原生 API**,直接对应 prefill/decode 分离、多阶段推理管线。如果我们的推理产品正在规划这个能力,**不应再自研上层编排**,而是:
   - 评估把我们的推理 CR(LLMInferenceService 之类)降级为 DisaggregatedSet 的 thin wrapper;
   - 或在 UI/平台层面直接暴露 DisaggregatedSet,类似我们之前对 LWS 的做法。
2. **API 起步即 v1、计划与 LWS 二进制合并**(issue [#789](https://github.com/kubernetes-sigs/lws/issues/789))—— 说明这是上游的"一等公民"路线,不是实验沙盒。架构层面可以按"会 GA"的预期下注。
3. **LWS fail-fast restart budget KEP 草稿**([#813](https://github.com/kubernetes-sigs/lws/pull/813))回应了大规模多节点推理的典型痛点:节点丢失后自愈 vs 立即失败。我们的多节点推理方案里这一块目前是靠探针/外部 controller,上游走通后可以削掉这层自定义。

### Kueue(企业级队列/公平调度)

1. **AFS + DRA 集成测试合并** 表明 Kueue 官方承诺"带 DRA 的 GPU 作业也能做公平分配",这是我们 AI 开发平台做"多租户 GPU 公平分享"最想要的组合,建议在内部 PoC 中提前跑通。
2. **Sandbox operator 集成** 把 Kueue 覆盖面从"训练/推理作业"扩展到"AI agent sandbox",如果我们在 agent 平台做租户隔离,这条路径可以直接复用 Kueue + pod 集成。
3. **MultiKueue 本轮连续两周都有重要 bug 修复**(断连、worker 驱逐、AdmissionCheck 匹配),稳定性在收敛,管理集群 + worker 集群分离的多集群 AI 平台拓扑可以更放心地上生产。
