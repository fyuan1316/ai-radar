# K8s AI 基础设施周报 2026-07-02

窗口:2026-06-25 -> 2026-07-02(7 天)。releases 部分与 2026-06-29 期有重叠,已标注"上期已报";PR/KEP 聚焦 06-29 之后的新动向。

## 摘要(3 条以内)
- **DRA 关键节点后移**:[KEP-5007 DRA Device Binding Conditions 的 GA 目标从 1.37 改到 1.38](https://github.com/kubernetes/enhancements/pull/6005)。设备绑定条件是 DRA 从"分配"走向"可依赖调度信号"的门槛,GA 推迟意味着我们把诊断/编排依赖 binding conditions 的时间表要顺延一个版本。
- **原生 disaggregated(P/D 分离)推理编排成形**:LWS 引入 [DisaggregatedSet 的 `slices` 字段](https://github.com/kubernetes-sigs/lws/pull/873),把 prefill/decode 整套角色拓扑作为一个可整体扩缩的单元(KEP 文档见 [#879](https://github.com/kubernetes-sigs/lws/pull/879))。这是 K8s 侧对标 vLLM P/D 分离部署的原生原语,直接关系我们推理平台的编排模型。
- **GPU Operator 在为 DRA 铺路**:新增 [node-labeling-controller(#2569)](https://github.com/NVIDIA/gpu-operator/pull/2569) 收敛所有节点打标逻辑,PR 明确写"当 DRA 集成进来时还会有一个控制器要打标";同时 [NVIDIADriver CRD 增加 upgradePolicy(#2582)](https://github.com/NVIDIA/gpu-operator/pull/2582) 支持按 CR 定制驱动升级策略。

## GPU / 加速器管理

- [NVIDIA GPU Operator #2569](https://github.com/NVIDIA/gpu-operator/pull/2569) — 新建 `node-labeling-controller`,把原先散在 clusterpolicy / nvidiadriver 两个控制器里的节点打标统一收口,并显式为未来 DRA 控制器打标预留位置。
  - 启示:这是 GPU Operator 架构从"device-plugin + ClusterPolicy"向"DRA-ready"过渡的信号。我们做多加速器纳管时,节点标签(`nvidia.com/*`、owner 标签、operand 开关)应设计成单一 reconcile 源,避免多控制器打标冲突——GPU Operator 正在替我们踩这个坑。
- [NVIDIA GPU Operator #2582](https://github.com/NVIDIA/gpu-operator/pull/2582) — `NVIDIADriver` CRD 增加 `upgradePolicy`,允许不同 NVIDIADriver CR 用不同的驱动升级策略;为空时回落到代码内默认(与 helm values 对齐)。
  - 启示:驱动升级策略从全局 helm 参数下沉到 per-CR,意味着同一集群里不同节点池可以有不同的滚动升级节奏(生产池保守、测试池激进)。我们的驱动生命周期管理应支持节点池粒度的升级策略,而不是集群一刀切。
- [NVIDIA GPU Operator #2527](https://github.com/NVIDIA/gpu-operator/pull/2527) — 驱动配置未变时原地重启驱动 Pod(与 compute-nv-watch 期重叠,此处从 K8s 编排视角记一笔)。
  - 启示:区分"整节点排空重装"与"仅重启"两条升级快慢路径,能显著降低驱动 bump 对在跑作业的打扰。我们的加速器运维要把"配置摘要一致 -> 快路径"这个判据纳入。
- [NVIDIA k8s-device-plugin #1788](https://github.com/NVIDIA/k8s-device-plugin/pull/1788) — 修 distributedAlloc tie-break,让 time-slicing 副本在物理 GPU 间均匀铺开(与 compute-nv-watch 重叠)。
  - 启示:软切分/time-slicing 的分配均衡直接影响单卡热点。我们评估共享 GPU 方案时要把"副本在物理设备上的分布"作为验收指标,而非只看总量。
- [NFD #2534](https://github.com/kubernetes-sigs/node-feature-discovery/pull/2534) — 修 nfd-master 在初始化失败时 `stop()` panic 与 informer 泄漏;[#2535](https://github.com/kubernetes-sigs/node-feature-discovery/pull/2535) 修 nfd-worker 对 `noPublishFeatures` 模式串未去空白导致匹配失效。
  - 启示:NFD 是所有 GPU/加速器特征发现的底座,它的控制器健壮性(初始化失败清理、配置解析容错)会直接影响节点是否被正确打标。生产平台应把 NFD 异常路径纳入巡检。

- 版本(上期已报,carryover):[GPU Operator v26.3.3](https://github.com/NVIDIA/gpu-operator/releases/tag/v26.3.3)(2026-06-25 patch)。本窗口 k8s-device-plugin / NFD 无新 release。

## 调度 & 资源分配

### Kueue

上期已报版本:[v0.18.2](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.18.2) / [v0.17.6](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.17.6)(2026-06-26)。本窗口 06-29 后合入一批修复,已 cherry-pick 到 0.17/0.18 分支,下个 patch 会带出:

- [#12644](https://github.com/kubernetes-sigs/kueue/pull/12644) — RayJob 默认 submitter pod template 丢失 admission 约束(node selector / toleration 等)的修复。
  - 启示:Ray 是我们分布式训练/推理的重点集成对象。Kueue 对 RayJob 的 admission 约束透传一旦有洞,submitter pod 可能落到错误节点池。RayJob e2e 要覆盖"约束是否完整传到所有子 Pod(head/worker/submitter)"。
- [#12671](https://github.com/kubernetes-sigs/kueue/pull/12671) — AFS(AdmissionFairSharing)ConsumedResources 的 CPU 在早期 reconcile/自重排时被截断为 0 的修复。
  - 启示:公平调度的资源计量若在时序边界出错,会导致配额分配失真。我们做多租户公平共享时要对"计量在 reconcile 竞态下的正确性"专门加压测。
- [#12647](https://github.com/kubernetes-sigs/kueue/pull/12647) — SparkApplication 的 node selector 在 driver 与 executor 之间互相覆盖的修复。
- [#12643](https://github.com/kubernetes-sigs/kueue/pull/12643) — MultiKueue 对 external-framework job 的远端元数据做 sanitize;[#12587](https://github.com/kubernetes-sigs/kueue/pull/12587) — 远端与本地抢占门(preemption gate)独立处理。
  - 启示:MultiKueue 走向生产会不断暴露"本地/远端状态语义混淆"类问题。多集群队列设计要严格区分本地对象与远端投影,元数据、抢占、autoscaler 信号都不能串。
- [#12045](https://github.com/kubernetes-sigs/kueue/pull/12045) — `ElasticJobsViaWorkloadSlices` 下 ElasticJobUngater 把所有 gated pod 无上限 ungate 的问题,改为按已授配额封顶。
  - 启示:弹性作业(workload slices)是 Kueue 对可伸缩训练/推理的新机制。ungate 不设上限会突破配额。我们若采用弹性作业,需验证"ungate 数量 <= 已 admit 配额"这个不变式。
- [#12659](https://github.com/kubernetes-sigs/kueue/pull/12659) — 为 `UnadmittedWorkloadObservability` feature gate 做前置的 workload 可观测流程。
  - 启示:未被 admit 的 workload 的可观测性(为什么排队、卡在哪)是排障刚需。我们的调度控制台应提前对齐这个 gate 的数据模型。

### DRA

- [KEP-5007 DRA Device Binding Conditions](https://github.com/kubernetes/enhancements/pull/6005) — GA 目标从 1.37 **改为 1.38**。这是本期最重要的时间表变化:上期我们记的是 1.37 GA,现已顺延。
  - 启示:设备绑定条件 GA 后,ResourceClaim/DeviceClaim 的绑定成功/等待/失败会成为可依赖的控制面信号。GA 推迟一个版本,意味着我们把"调度诊断读 binding conditions 而非只看 Pod Pending event"的落地节点相应后移,近期仍需保留基于 event 的兜底诊断。
- 本窗口 [dra-evolution](https://github.com/kubernetes-sigs/dra-evolution) 与 [scheduler-plugins](https://github.com/kubernetes-sigs/scheduler-plugins) 无 merged PR / release。DRA 的动作目前集中在 k/enhancements(KEP 层)和上游 k/k 实现,设备层演进仓相对静默。

### LWS / JobSet

- [LWS #873](https://github.com/kubernetes-sigs/lws/pull/873)(实现,open)+ [#879](https://github.com/kubernetes-sigs/lws/pull/879)(KEP 文档,已合)— 为 `DisaggregatedSet` 增加 `spec.slices`:把一整套角色拓扑(如 prefill + decode)作为一个"切片"整体复制成 N 份,每份独立滚动。切片标签 `disaggregatedset.x-k8s.io/slice`,命名 `<ds>-<slice>-<revision>-<role>`;`slices` 是纯扩缩旋钮,改它不触发滚动。设计上刻意用 "slices" 而非 "replicas" 以免和 LWS replica 混淆,并为后续 per-domain 放置策略(#848)打基础。
  - 启示:这是 K8s 原生对标 vLLM/推理引擎 P/D 分离部署的编排原语——过去要跑 K 份相同 P/D 组合得复制 YAML,现在一个字段搞定。我们的推理平台如果做 disaggregated serving,应直接对齐 DisaggregatedSet 的模型(slice 为扩缩单位、role 为 P/D 角色),而不是自造一套 CRD。这也解释了为什么 KServe/OAI 系持续押注 LWS 作为多节点/多角色推理底座。
- [JobSet #1249](https://github.com/kubernetes-sigs/jobset/pull/1249) — 仅 gomega 依赖 bump,无功能变化。

## KEP 动向

- [KEP-5007 DRA Device Binding Conditions](https://github.com/kubernetes/enhancements/pull/6005) — GA 目标 1.37 -> 1.38(见上,DRA 段)。
- [KEP-5682 PDB for Multi-Pod Replicas](https://github.com/kubernetes/enhancements/pull/5671) — 为"一个逻辑副本 = 多个 Pod"的工作负载(正是 LWS / 多节点推理的形态)提供 PodDisruptionBudget 语义。
  - 启示:多节点推理/训练里,一个副本跨多 Pod,现有 PDB 按单 Pod 计算会误伤。这个 KEP 落地后,我们能对 LWS 组、gang 组做整组级别的中断保护,是生产可用性的关键补齐,值得重点跟踪。
- [KEP-4438 Restarting sidecar containers during Pod termination](https://github.com/kubernetes/enhancements/pull/6218) — 将 alpha 文档与已合入 k/k 的实现对齐。
  - 启示:AI Pod 普遍带 sidecar(指标、日志、驱动/toolkit 注入)。终止阶段 sidecar 的重启行为会影响优雅退出与数据落盘。做长训练作业的 checkpoint/退出流程时要留意 sidecar 生命周期语义变化。
- [KEP-5972 Dynamic Containers](https://github.com/kubernetes/enhancements/pull/6169)(sig-node,可变容器)与 [KEP-5683 Specialized Lifecycle Management](https://github.com/kubernetes/enhancements/pull/5769) — Pod/容器生命周期正在变得更可变、更精细。
  - 启示:这类 sig-node 能力长期会影响"能否在不重建 Pod 的前提下调整 AI 工作负载"(如原地换镜像、调参)。与在线资源 resize、原地驱动重启是同一趋势线,先记录跟踪。
- 关注但本窗口无确凿 PR 活动:gang scheduling 方向(scheduler-plugins 的 PodGroup 与 KEP-4671 gang 语义收敛)。DRA + gang 是我们 GPU 调度选型的两条主线,持续盯 scheduler-plugins 与 k/enhancements。

## 值得跟进
- [ ] 因 KEP-5007 GA 顺延到 1.38,更新我们 DRA 依赖路线图:短期保留基于 Pod event 的调度诊断兜底,binding conditions 作为 1.38 后的增强。
- [ ] 评估 LWS DisaggregatedSet(slices + P/D 角色)作为我们 disaggregated 推理编排的直接底座,对比自研 CRD 的取舍;跟踪 per-domain 放置策略 #848。
- [ ] 跟 GPU Operator node-labeling-controller 收敛节点打标的做法,审视我方多加速器纳管的标签管理是否有多控制器打标冲突风险,并预留 DRA 打标位。
- [ ] 若采用 Kueue 弹性作业(WorkloadSlices),验证 ungate 数量受配额封顶(参考 #12045)、RayJob admission 约束完整透传(#12644)。
- [ ] 跟踪 KEP-5682 PDB for Multi-Pod Replicas,规划 LWS/gang 组的整组中断保护。
