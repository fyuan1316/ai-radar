# K8s AI 基础设施周报 2026-06-29

窗口:2026-06-22 -> 2026-06-29(7 天)

## 摘要(3 条以内)
- [Kueue v0.18.2](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.18.2) 与 [v0.17.6](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.17.6) 同日发布，重点修复 DRA workload admission、MultiKueue/Cluster Autoscaler 误触发、Ray GCS Redis cleanup quota accounting 等问题。
- [KEP-5981 DRA Sharing Affinity for Consumable Capacity](https://github.com/kubernetes/enhancements/pull/5987) 和 [KEP-5007 DRA Device Binding Conditions GA in 1.37](https://github.com/kubernetes/enhancements/pull/6005) 本周更新，DRA 正从“设备分配 API”进入共享容量、绑定条件、调度语义细化阶段。
- NVIDIA 侧 [GPU Operator v26.3.3](https://github.com/NVIDIA/gpu-operator/releases/tag/v26.3.3) 与 [k8s-device-plugin v0.19.3](https://github.com/NVIDIA/k8s-device-plugin/releases/tag/v0.19.3) 都有 patch，设备管理栈仍需按 driver/runtime/device-plugin 组合验证。

## GPU / 加速器管理

- [NVIDIA GPU Operator v26.3.3](https://github.com/NVIDIA/gpu-operator/releases/tag/v26.3.3) — 本周 patch release，官方 release 指向 NVIDIA GPU Operator release notes。
  - 启示:GPU Operator 企业交付的风险仍在驱动、container toolkit、device plugin、MIG、DCGM、节点 OS 的组合矩阵。我们的兼容性测试应按 operator patch 版本重跑 smoke，而不是只锁 minor。
- [NVIDIA k8s-device-plugin v0.19.3](https://github.com/NVIDIA/k8s-device-plugin/releases/tag/v0.19.3) — 回滚 MOFED/GDRCopy 默认启用相关特性，并升级 Go / `x/net`。
  - 启示:高速网络/RDMA/GDRCopy 这类能力默认开启会影响集群稳定性，生产平台应把这些能力做成显式 profile，并在节点标签、RuntimeClass、device plugin 参数之间保持一致。

## 调度 & 资源分配

### Kueue

- [Kueue v0.18.2](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.18.2) — DRA workload 带 `matchAttribute` 或 device config 时不再被误判 unsupported，而是进入 quota/admission 流程。
  - 启示:DRA 与队列配额的结合已经开始处理真实设备约束。我们验证 GPU/NPU DRA driver 时，必须把 ResourceClaim 参数、quota accounting、admission error path 放进同一套 e2e。
- [Kueue v0.18.2](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.18.2) — MultiKueue 修复 manager cluster 中 admitted Pod 误触发 Cluster Autoscaler 扩容的问题。
  - 启示:多集群 AI 训练/推理队列里，manager 集群对象必须避免污染本地 autoscaler 信号。我们的多集群调度设计要区分“占位对象”和“真实可调度 Pod”。
- [Kueue PR #12612](https://github.com/kubernetes-sigs/kueue/pull/12612) — 修复 MultiKueue remote client reconnect data race。
  - 启示:MultiKueue 进入生产会遇到远端连接抖动、状态竞争、重复作业等问题。多集群队列的控制面稳定性要按网络不稳定场景压测。
- [Kueue PR #12614](https://github.com/kubernetes-sigs/kueue/pull/12614) 和 [#12613](https://github.com/kubernetes-sigs/kueue/pull/12613) — KueueViz 修复 websocket OOM 风险和 flavor empty resources crash。
  - 启示:队列可视化不是附属功能，dashboard 本身也要做资源限制和异常数据保护。我们的调度控制台要避免大规模 workload/queue 下成为新的风险点。

### DRA

- [KEP-5981](https://github.com/kubernetes/enhancements/pull/5987) — DRA Sharing Affinity for Consumable Capacity。
  - 启示:DRA 正在补“共享容量 + 亲和”的语义，这会直接影响 vGPU/vNPU、MIG、显存/带宽切分等能力是否能原生表达。我们应把现有共享设备模型映射到 consumable capacity，而不是继续只靠 extended resource。
- [KEP-5007](https://github.com/kubernetes/enhancements/pull/6005) — DRA Device Binding Conditions 计划在 Kubernetes 1.37 GA。
  - 启示:设备绑定条件 GA 后，DRA 设备分配失败/等待/绑定状态会成为可依赖的控制面信号。我们的调度诊断应开始读取 ResourceClaim/DeviceClaim 条件，而不是只看 Pod Pending event。
- [KEP-6007](https://github.com/kubernetes/enhancements/pull/6008) — Topology Manager 增加 single-numa-node workload density 选项。
  - 启示:AI 节点的 NUMA/拓扑亲和不是只在 device plugin 层解决，kubelet Topology Manager 选项会影响高密度单 NUMA 部署。GPU/NPU 共享场景需要把 topology policy 纳入节点池 profile。

### LWS / JobSet

- 本窗口未确认到 LWS / JobSet 新 release。KServe/ODH 和 OpenFuyao 都在把 LWS 作为多副本/多 DP 推理编排基础，建议继续跟踪 LWS 与 KServe LLMISVC、Kueue 的集成边界。

## KEP 动向

- [KEP-3953 In-place Node Resource Resize](https://github.com/kubernetes/enhancements/pull/3955) 本周仍有更新。
  - 启示:节点资源可在线调整对 AI 节点池长期有价值，但它改变的是 node allocatable/capacity 语义。结合 GPU/NPU 设备时，要特别验证 kubelet、scheduler cache、autoscaler 的一致性。
