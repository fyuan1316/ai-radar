# K8s AI 基础设施周报 2026-06-11

> 覆盖窗口:过去 7 天(2026-06-04 ~ 06-11)。本期无任何仓库发布新 release,信号集中在 KEP 推进与 PR 层面。

## 摘要(3 条以内)

1. **DRA 设备共享进入 v1.37 beta 冲刺**:KEP-5075「DRA Consumable Capacity」目标阶段从 v1.36 beta 顺延到 **v1.37 beta**,这是用原生 DRA 实现「一块设备按容量切给多个互不相关的 ResourceClaim」的核心机制——也就是 GPU 显存/算力分片的官方路线,直接对标 HAMi/MIG 这类厂商方案。
2. **原生 Gang 调度落到上层编排**:JobSet 新增了对接 Workload Aware Scheduling(`scheduling.k8s.io/v1alpha2`,K8s 1.36+ alpha)的完整文档,演示用原生 Workload/PodGroup 做 gang scheduling 与 PodGroup 级抢占——上游正把 gang 调度从 scheduler-plugins 的 coscheduling 往 kube-scheduler 原生 API 迁。
3. **LWS 向「分离式推理」演进**:LWS 围绕 DisaggregatedSet(prefill/decode 角色分离)做资产迁移与 rollout 等待逻辑收紧,并起草了 fail-fast 重启预算 KEP——多角色 LLM 推理拓扑正在成为 LWS 一等公民。

## GPU / 加速器管理

本周 NVIDIA 侧的实质改动与 `compute-nv-watch` 同源,从 AI 基础设施视角再记一笔:

- **特性默认值回退 + 按内核模块推断**:k8s-device-plugin 把 gds / gdrcopy / mofed 三个能力 flag 的默认值从 true 回退为 false(https://github.com/NVIDIA/k8s-device-plugin/pull/1837 ,后续 https://github.com/NVIDIA/k8s-device-plugin/pull/1841 );gpu-operator 配套改为在 state-device-plugin 启动前用 lsmod 探测 gdrdrv / nvidia_fs 内核模块、按宿主实际加载情况注入对应 envar(https://github.com/NVIDIA/gpu-operator/pull/2525 ,release-26.3 backport https://github.com/NVIDIA/gpu-operator/pull/2528 )。能力开启从「插件侧默认全开」转为「算子侧按宿主实测推断」。
  - 启示:这是把「设备能力发现」的真相源从声明式默认值挪到运行时探测,避免在没装 GDS/GDRCopy 内核模块的节点上误开能力导致 Pod 起不来。我们若有类似的加速器能力开关,应学这套「探测优先、默认保守」的策略,而不是默认全开。
- **NRI 模式不再挂 containerd 运行时配置**:gpu-operator 在启用 NRI 时丢弃 containerd config 的 hostPath 挂载(https://github.com/NVIDIA/gpu-operator/pull/2514 ),以兼容 Talos 这类只读宿主。
- **NFD 无新合并 PR**:`kubernetes-sigs/node-feature-discovery` 本窗口无 release、无 merged PR。

## 调度 & 资源分配

### Kueue

DRA 集成持续硬化,且 0.17 / 0.18 两条 release 分支都在积极回合:

- **DRA reconcile 错误区分「可重试 / 确定性失败」**(https://github.com/kubernetes-sigs/kueue/pull/12002 ):此前所有 DRA 解析错误都走 controller-runtime 退避重试,导致「不支持的 DRA 特性、未映射的 DeviceClass、非整数扩展资源」这类永久性错误被无意义地反复重试。现在内部/API 失败与集群侧设备短缺仍保留退避(等 ResourceSlice 变化后能恢复),而 spec 配置类永久错误直接让 workload 进入 inadmissible 不再 requeue。取代了被回滚的固定速率重试方案 #11943。
- **DeviceClass 删除后挂起 workload 不再卡死**(https://github.com/kubernetes-sigs/kueue/pull/11929 ,已 backport 到 release-0.18 https://github.com/kubernetes-sigs/kueue/pull/12093 ):删除 DeviceClass 后挂起的 workload 现在会被正确重新入队。
- **Topology-Aware Scheduling(TAS)文档成体系**:新增 TAS 任务页(https://github.com/kubernetes-sigs/kueue/pull/11259 )与 TASReplaceNodeOnNodeTaints 文档(https://github.com/kubernetes-sigs/kueue/pull/11967 ),TAS 正从特性走向可被运维直接使用的能力。
- 启示:Kueue 的 DRA 路径正在从「能跑」走向「错误语义清晰、可运维」。如果我们的配额/队列层准备接 DRA,这两个 PR 是很好的错误分类参考——别把配置错误当瞬时错误无限重试。

### DRA(本期重中之重)

两个 sig-scheduling 的 DRA KEP 在本窗口推进:

- **KEP-5075「DRA Consumable Capacity」→ 目标 v1.37 beta**(stage 已置 beta,latest-milestone v1.37;alpha 落在 v1.34)。核心:让相互独立、甚至跨 namespace 的多个 ResourceClaim 各自从同一块底层设备的总容量里「切一份」,带 `requestPolicy`(每 claim 最小量等)与容量感知调度,allocation result 里新增「已消耗容量」字段;并引入 distinct attribute 约束防止同一 claim 重复分配同一块多可分配设备。与分区设备(KEP-4815)、网络设备共享(KEP-4817)、PrioritizedList(KEP-4816)联动。源:https://github.com/kubernetes/enhancements/blob/master/keps/sig-scheduling/5075-dra-consumable-capacity/README.md
- **KEP-5491「DRA: List Types for Attributes」→ v1.36 alpha,v1.37 细化**(detailed design 与 graduation criteria 更新,https://github.com/kubernetes/enhancements/pull/6131 )。让 ResourceSlice 的 attribute 支持「列表类型」以表达多 PCIe root / 多 NUMA 节点这类拓扑关系,并重定义 ResourceClaim 的 `matchAttribute / distinctAttribute` 约束语义同时兼容列表与标量。源:https://github.com/kubernetes/enhancements/blob/master/keps/sig-scheduling/5491-dra-list-types-for-attributes/README.md
- 启示:Consumable Capacity 是「原生 DRA 做 GPU 分片」最关键的一块拼图——它把 HAMi-core / MIG 那种「一卡多租」的语义搬进了 K8s 标准 API。一旦 v1.37 进 beta,设备分片就有了不依赖厂商 device-plugin 的标准做法,这会重塑我们对 GPU 共享方案的选型(是继续绑厂商 DRA driver,还是押注标准容量模型)。List Types 则是表达 NUMA/PCIe 亲和的基础,和拓扑感知调度直接相关。**建议本季度专门读一遍 5075 的 API 设计**。
- `kubernetes-sigs/dra-evolution`:本窗口无 merged PR。

### LWS / JobSet

- **LWS:DisaggregatedSet(prefill/decode 分离式推理)重构进行中**
  - 迁移 DisaggregatedSet 辅助资产(plan-steps、fixture 测试、三角色 v1 样例)为后续删除独立 `disaggregatedset/` 目录做准备(https://github.com/kubernetes-sigs/lws/pull/875 ,关联 issue #789),信号是该能力正从独立子树并入 LWS 主体。
  - 收紧 `ForSingleActiveRevision` 等待逻辑(https://github.com/kubernetes-sigs/lws/pull/870 ):修复滚动更新中途(新 revision 只有 prefill=1/decode=0 时)误判 rollout 已完成的两个逃逸窗口。
  - **KEP 草案:fail-fast 重启预算 + init 阶段 peer DNS**(https://github.com/kubernetes-sigs/lws/pull/813 ):opt-in、默认 nil、不改现有行为,新增终止性 LWS condition;主诉求是「快速失败重启预算」,init-phase DNS 为次要目标。关联 kubeflow/trainer。
  - e2e 基线升到 K8s 1.36(https://github.com/kubernetes-sigs/lws/pull/874 )。
- **JobSet:对接原生 Workload Aware Scheduling 文档**(https://github.com/kubernetes-sigs/jobset/pull/1241 ):新增顶层文档,演示用 `scheduling.k8s.io/v1alpha2` 的 Workload + PodGroup 资源给 JobSet 做 gang scheduling,以及基于 PriorityClass + disruptionMode 的 PodGroup 级 gang 抢占;前置条件 K8s 1.36+、alpha feature gate。
- 启示:两条线都指向「原生化」——LWS 把分离式 LLM 推理(prefill/decode 拆角色)做成原生拓扑;JobSet 把 gang 调度从 scheduler-plugins coscheduling 迁到 kube-scheduler 原生 Workload API。我们若在产品里自带 coscheduling 插件,要开始评估 1.36+ 原生 Workload/PodGroup 的迁移路径。
