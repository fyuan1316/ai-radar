# K8s AI 基础设施周报 2026-07-16

窗口:2026-07-09 -> 2026-07-16(7 天)

> 说明:本窗口内的 release(NFD v0.19.0、Kueue v0.18.3/v0.17.7,均 07-10 发布)已在 [2026-07-13 期](https://github.com/fyuan1316/ai-radar/blob/main/digests/2026-07-13-k8s-ai-infra.md)覆盖,本期聚焦 07-13 之后新合入的 PR 与 KEP 动向,不重复列旧 release。

## 摘要(3 条以内)
- Kueue 本周主线是 **TAS(拓扑感知调度)硬化**:把"固定 30s NotReady 即判节点失败"改成仅在 Pod 真正终止时才替换节点([#13043](https://github.com/kubernetes-sigs/kueue/pull/13043)),并在 API 层拒绝负 subGroupCount、稳定域排序、补 SkipReassignmentForPodOwnedWorkloads 开关——多节点训练/推理的 gang 放置正在从"能跑"走向"账不算错"。
- Kueue 同时清理 feature gate 走向成熟:[ManagedJobsNamespaceSelectorAlwaysRespected 升 GA](https://github.com/kubernetes-sigs/kueue/pull/13021)、[移除 LendingLimit gate](https://github.com/kubernetes-sigs/kueue/pull/13067),并为 0.19 的 **WAS(工作负载感知调度)**接入上游 scheduler-library 打地基([#13045](https://github.com/kubernetes-sigs/kueue/pull/13045))。
- 上游 KEP 两条与我们直接相关:[KEP-5710 Workload-aware preemption](https://github.com/kubernetes/enhancements/issues/5710) 补 PodGroupPreemptionPolicy 独立 feature gate,[KEP-5732 Topology-aware workload scheduling](https://github.com/kubernetes/enhancements/issues/5732) 推进——in-tree 主调度器的 gang/拓扑放置能力成型,将与 Kueue 的 TAS/WAS 直接对接。

## GPU / 加速器管理

- [NVIDIA GPU Operator PR #2407](https://github.com/NVIDIA/gpu-operator/pull/2407) — 彻底停止处理 kata manager,该 operand 标记 deprecated,ClusterPolicy 里对应段落的改动一律不再响应,并从 state manager 移除。
  - 启示:GPU operator 在做 operand 级瘦身。我们维护加速器 operator 时,要有明确的 operand 生命周期(active/deprecated/removed)和"对已废弃字段不 reconcile"的契约,避免旧 CR 字段引发无意义 reconcile 或回滚。
- [NVIDIA GPU Operator PR #2607](https://github.com/NVIDIA/gpu-operator/pull/2607) — 新增 deploy label 管理第三方 GPU client pod(07-13 期已预告,本周合入)。
  - 启示:延续"operator 承接第三方 GPU client 编排"的方向,需在我们产品里划清 operator 托管 vs 平台/用户托管的边界。
- [NVIDIA GPU Operator PR #2637](https://github.com/NVIDIA/gpu-operator/pull/2637) — 打包默认 dcgm-exporter 升到 v4.6.0-4.8.3。
  - 启示:与 [compute-nv-watch 07-16 期](https://github.com/fyuan1316/ai-radar/blob/main/digests/2026-07-16-compute-nv-watch.md)呼应(该版补齐 DRA ResourceSlice/MIG 监控,并有 Hostname→hostname、NVLink counter→gauge 破坏性口径变更)。operator 默认镜像跟进后,监控看板/告警口径要同步核对。
- [NVIDIA k8s-device-plugin PR #1923](https://github.com/NVIDIA/k8s-device-plugin/pull/1923) — bump nvidia-container-toolkit 到 v1.20.0-rc.1。
- [NVIDIA k8s-device-plugin PR #1922](https://github.com/NVIDIA/k8s-device-plugin/pull/1922) — NFD chart 升 v0.19.0 并切换到 OCI artifacts 分发。
  - 启示:NFD 依赖切 OCI artifact 是分发方式变化,离线/内网环境的镜像仓库同步策略要覆盖 OCI chart,不能只镜像传统 helm repo。

## 调度 & 资源分配

### Kueue

- [PR #13043](https://github.com/kubernetes-sigs/kueue/pull/13043) — **TAS:仅在 Pod 终止时替换节点,废弃固定时长 NotReady 判失败**。原逻辑到 NodeFailureDelay(30s)就把节点标失败,但节点可能随后恢复且 Pod 仍在其上运行,导致重算的拓扑分配与 Pod 实际占用的节点背离、双向污染 per-node 容量核算(真实节点被当空闲而超额准入)。
  - 启示:这是拓扑调度落地里最容易踩的坑——"节点短暂 NotReady ≠ 作业迁走"。我们做多节点训练/推理的拓扑放置时,节点故障判定必须与 Pod 实际生命周期挂钩,而不是固定超时,否则容量账会两头错。
- [PR #13061](https://github.com/kubernetes-sigs/kueue/pull/13061) / [#13031](https://github.com/kubernetes-sigs/kueue/pull/13031) / [#13017](https://github.com/kubernetes-sigs/kueue/pull/13017) — TAS 在 API 层拒绝负 subGroupCount、稳定域排序保证打分平手时放置确定性、新增 SkipReassignmentForPodOwnedWorkloads feature gate。
  - 启示:TAS 正从功能补齐转向确定性与健壮性。拓扑调度的"可复现放置"是排障和容量规划的前提,我们自研调度也要保证同输入同放置。
- [PR #13021](https://github.com/kubernetes-sigs/kueue/pull/13021) — **ManagedJobsNamespaceSelectorAlwaysRespected 升 GA**。
  - 启示:多租户下"哪些命名空间的作业强制受 Kueue 管控"成为稳定能力。我们的队列产品应把 namespace 级强制纳管作为默认合规基线,而非可选项。
- [PR #13067](https://github.com/kubernetes-sigs/kueue/pull/13067) — 移除 LendingLimit feature gate(能力沉淀为默认)。
  - 启示:ClusterQueue 间借还配额(cohort lending)已稳定。做集群级 GPU/NPU 池的公平共享时,lending/borrowing 可作为既定能力设计,不必再当实验特性。
- [PR #13045](https://github.com/kubernetes-sigs/kueue/pull/13045) — **WAS(工作负载感知调度)接入上游 scheduler-library 依赖**,为 0.19 的最小集成铺路(part of #8871 / #12256)。
  - 启示:Kueue 与 in-tree 主调度器(KEP-5710/5732 的 workload-aware/topology 放置)正在收敛为一套。我们的队列/调度选型要盯这条融合线,避免自建一套与上游 gang/拓扑语义分叉的调度层。
- [PR #13047](https://github.com/kubernetes-sigs/kueue/pull/13047) — MultiKueue:远端以 OutOfSync 结束时改为重试,而非把同步失败当作业正常完成(此前会让 manager 侧 Job 悬挂:suspend=false 无 Pod、无信号)。
  - 启示:多集群队列里"远端 worker 的 mutating webhook 改了 pod template 导致 workload 与预建对象背离"是真实故障路径。我们做多集群 GPU 池时,同步失败必须走重试/告警,不能静默视为成功。
- [PR #13029](https://github.com/kubernetes-sigs/kueue/pull/13029) — 给 visibility pending-workloads 接口加上限,防止大量 pending workload 触发 OOM/panic;[#13049](https://github.com/kubernetes-sigs/kueue/pull/13049) 让 visibility auth-reader RoleBinding 可选。
  - 启示:队列可观测接口在做稳定性与最小权限收敛。我们暴露"为什么作业没跑"的 pending 列表时要分页限流,并让 RBAC 依赖可裁剪。
- [PR #13071](https://github.com/kubernetes-sigs/kueue/pull/13071)(及 release-0.17/0.18 backport [#13082](https://github.com/kubernetes-sigs/kueue/pull/13082)/[#13081](https://github.com/kubernetes-sigs/kueue/pull/13081))— workload slicing 拒绝跨命名空间替换。
  - 启示:多租户隔离的边界 bug。workload slicing(弹性伸缩切片)必须校验命名空间归属,我们复用该能力时要把跨租户替换作为默认拒绝项测试。
- [PR #13044](https://github.com/kubernetes-sigs/kueue/pull/13044)(backport [#13070](https://github.com/kubernetes-sigs/kueue/pull/13070)/[#13060](https://github.com/kubernetes-sigs/kueue/pull/13060))— 减少 reclaimable pods 以防配额泄漏。
  - 启示:配额泄漏会让集群"看着满了其实空着"。生产引入 Kueue 要把 reclaimable/quota 一致性纳入回归。

### DRA

- 本窗口 DRA 无独立进展:`kubernetes-sigs/dra-evolution` 无新提交,Kueue 侧仅 [#13038](https://github.com/kubernetes-sigs/kueue/pull/13038)(counter-based DRA 资源以 BinarySI 单位展示)向 release-0.18 的 cherry-pick(原 #12989 已在 07-13 期覆盖)。
  - 启示:DRA 的调度侧样板已基本就位,本周重心转到 Kueue TAS/WAS 与主调度器的对接。DRA 端到端 smoke(DeviceClass -> ResourceClaimTemplate -> ResourceSlice -> CDI -> workload)仍是我们要自建的验证项,上游本周没有替我们做掉。

### LWS / JobSet

- [LWS PR #929](https://github.com/kubernetes-sigs/lws/pull/929) — 升 golang.org/x/text 到 v0.39.0 修复 CVE-2026-56852。
  - 启示:多节点推理底座组件的传递依赖 CVE 要跟。我们若基于 LWS 打产品镜像,需把该 CVE 纳入 SBOM/扫描基线。
- [JobSet PR #1276](https://github.com/kubernetes-sigs/jobset/pull/1276) — webhook 忽略处于终止态的 leader pod,防止重试时死锁。
  - 启示:大规模训练作业编排里,webhook 对终止态 pod 的处理不当会造成 replicated job 重试死锁。我们做作业编排的准入校验要对 terminal pod 显式短路。

## KEP 动向

> 边界说明:以下 KEP 是 in-tree 主调度器能力,[k8s-core 07-16 期](https://github.com/fyuan1316/ai-radar/blob/main/digests/2026-07-16-k8s-core.md)已从"通用调度器"角度记录 v1.37 alpha 的 PodGroup/workload-aware scheduling;此处补记它们对 AI 作业(gang/拓扑)的直接影响。

- [KEP-5710 Workload-aware preemption](https://github.com/kubernetes/enhancements/issues/5710) — [PR #6222](https://github.com/kubernetes/enhancements/pull/6222) 补充 PodGroupPreemptionPolicy 描述以对齐实现。因 alpha 实现漏了该字段,它将滞后一个 release,需独立 feature gate 以免拖慢毕业。
  - 启示:主调度器原生 PodGroup(gang)现在有了"成组抢占策略"。这决定我们未来是继续用 scheduler-plugins coscheduling / Kueue,还是切到 in-tree PodGroup。抢占策略(整组抢 vs 部分抢)直接影响训练作业的 checkpoint 与最小运行收益,要提前建产品映射。
- [KEP-5732 Topology-aware workload scheduling](https://github.com/kubernetes/enhancements/issues/5732) — [PR #6234](https://github.com/kubernetes/enhancements/pull/6234) 重命名 scheduler_generated_placements_total 指标。
  - 启示:主调度器要原生输出"生成了哪些拓扑放置"的可观测指标。这与 Kueue TAS 是同一诉求的两端,我们的调度可观测口径应向该指标名对齐,避免自造一套。

## 值得跟进
- [ ] 跟踪 Kueue 0.19 的 WAS/scheduler-library 集成([#12256](https://github.com/kubernetes-sigs/kueue/pull/12256)、[issue #8871](https://github.com/kubernetes-sigs/kueue/issues/8871)):判断我们调度层是接上游 workload-aware,还是维持自研,这是选型岔路口。
- [ ] 把"节点故障判定挂 Pod 生命周期而非固定超时"([#13043](https://github.com/kubernetes-sigs/kueue/pull/13043))纳入我们多节点训练调度的容量核算设计,并做超时误判的回归用例。
- [ ] 评估 in-tree PodGroup 抢占(KEP-5710)vs scheduler-plugins coscheduling vs Kueue 三条 gang 路线,给出 6 个月选型建议。
- [ ] 将 NFD chart 切 OCI artifacts([#1922](https://github.com/NVIDIA/k8s-device-plugin/pull/1922))纳入离线/内网分发方案;dcgm-exporter v4.6+ 的破坏性 label 口径变更同步监控看板。
- [ ] Kueue 多租户隔离回归:跨命名空间 workload slicing 拒绝([#13071](https://github.com/kubernetes-sigs/kueue/pull/13071))、visibility 接口限流与 RBAC 可裁剪([#13029](https://github.com/kubernetes-sigs/kueue/pull/13029)/[#13049](https://github.com/kubernetes-sigs/kueue/pull/13049))。
