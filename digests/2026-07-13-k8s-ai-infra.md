# K8s AI 基础设施周报 2026-07-13

窗口:2026-07-06 -> 2026-07-13(7 天)

## 摘要(3 条以内)
- [Kueue v0.18.3](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.18.3) 与 [v0.17.7](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.17.7) 本周发布，同时设计层继续推进 preemption protection、ClusterQueue 默认最长执行时间、MultiKueue cross-cluster preemption、DRA resource formatting。
- LWS 的 DisaggregatedSet 正快速补齐推理拓扑能力:placement policy、per-role autoscaling、topology-aware scheduling、CompositePodGroup integration 都在本周活跃。
- NVIDIA GPU Operator 开始出现 DRA-based stack 的 `GPUCluster` CRD/controller，同时 NFD v0.19.0 发布并推进 feature allow/deny list、network topology updater、image compatibility scheduler。

## GPU / 加速器管理

- [NVIDIA GPU Operator PR #2571](https://github.com/NVIDIA/gpu-operator/pull/2571) — 增加 DRA-based stack 的 `GPUCluster` CRD 和 controller。
  - 启示:NVIDIA operator 侧开始给 DRA 栈单独建控制面对象。我们设计 GPU/NPU operator 时，要准备从 ClusterPolicy/device-plugin 模式迁移到 DRA profile/cluster 对象，尤其是设备发现、ResourceSlice、CDI、驱动生命周期之间的关系。
- [NVIDIA GPU Operator PR #2572](https://github.com/NVIDIA/gpu-operator/pull/2572) — 支持没有 ClusterPolicy 时 reconcile NVIDIADriver。
  - 启示:驱动生命周期正在从“大一统 ClusterPolicy”解耦。客户只想装 driver、只想装 DRA plugin、只想装 DCGM 的场景会更多，operator 需要组件级启停。
- [NVIDIA GPU Operator PR #2607](https://github.com/NVIDIA/gpu-operator/pull/2607) — 增加 deploy label 管理 third-party GPU client pods。
  - 启示:GPU operator 逐渐承担第三方 GPU client 生命周期编排。我们的加速器 operator 也要明确哪些客户端 pod 由 operator 管，哪些交给平台插件或用户应用管。
- [NVIDIA k8s-device-plugin PR #1899](https://github.com/NVIDIA/k8s-device-plugin/pull/1899) — MIG device placement 无法确定时暴露 NVML error。
  - 启示:MIG/vGPU/切分场景的错误必须能从设备层直达用户事件和平台告警，不能只在 device-plugin 日志里。
- [NVIDIA k8s-device-plugin PR #1912](https://github.com/NVIDIA/k8s-device-plugin/pull/1912) — vGPU vendor capability record 异常时跳过而不是 panic。
  - 启示:vGPU 能力发现链路需要容错，尤其在客户混用不同 vGPU 版本、driver 分支和虚拟化栈时。

## 调度 & 资源分配

### Kueue

- [Kueue v0.18.3](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.18.3) / [v0.17.7](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.17.7) — 本周发布 patch 版本。
  - 启示:Kueue 仍在高频修复窗口。生产环境引入 Kueue 时，不能只看 minor，需要跟踪 patch release 和回归矩阵，尤其 MultiKueue、preemption、RayCluster 等路径。
- [Kueue PR #12998](https://github.com/kubernetes-sigs/kueue/pull/12998) — KEP:Pluggable distributed tracing(OpenTelemetry)。
  - 启示:批任务队列的调度决策会进入 trace 体系。我们的队列/调度产品应输出 admission、preemption、quota、flavor assignment 的可观测链路，便于解释“为什么作业没跑”。
- [Kueue PR #10596](https://github.com/kubernetes-sigs/kueue/pull/10596) 与 [#10479](https://github.com/kubernetes-sigs/kueue/pull/10479) — ClusterQueue default maximum execution time。
  - 启示:队列层开始内置最长执行时间控制，这会影响长训练任务、Notebook 占坑和共享集群公平性。我们需要给不同队列提供默认 runtime policy，而不是只靠用户手动 TTL。
- [Kueue PR #12996](https://github.com/kubernetes-sigs/kueue/pull/12996) / [#12791](https://github.com/kubernetes-sigs/kueue/pull/12791) — Preemption protection，保证最小运行时间。
  - 启示:AI 作业 preemption 不能只按优先级立即抢占，还要考虑 checkpoint 周期和最小运行收益。我们的抢占策略需要接入训练框架 checkpoint 信号。
- [Kueue PR #11376](https://github.com/kubernetes-sigs/kueue/pull/11376) — MultiKueue Cross-Cluster Preemption(alpha)。
  - 启示:多集群队列的抢占语义正在形成。我们做多集群 GPU/NPU 池时，需要定义跨集群抢占、回收、迁移和用户可见事件。
- [Kueue PR #12989](https://github.com/kubernetes-sigs/kueue/pull/12989) — counter-based DRA resources 使用 BinarySI units 展示。
  - 启示:DRA 资源展示会直接影响用户理解和配额报表。切片、显存、带宽、token cache 这类非传统资源要有一致单位。

### DRA

- 本轮 `kubernetes-sigs/dra-evolution` repo pulls 端点返回 404，未能可靠抓取该源。
- [Kueue PR #12989](https://github.com/kubernetes-sigs/kueue/pull/12989)、[NVIDIA GPU Operator PR #2571](https://github.com/NVIDIA/gpu-operator/pull/2571)、[KServe DRA 示例 PR #5642](https://github.com/kserve/kserve/pull/5642) 共同说明:DRA 已从 K8s API 讨论进入调度、operator、模型服务样例的贯通阶段。
  - 启示:产品层要尽快建立 DRA 端到端 smoke:DeviceClass -> ResourceClaimTemplate -> ResourceSlice -> CDI -> 推理/训练 workload，而不是只验证 device-plugin limit。

### LWS / JobSet

- [LWS PR #916](https://github.com/kubernetes-sigs/lws/pull/916) 与 [#906](https://github.com/kubernetes-sigs/lws/pull/906) — DisaggregatedSet 增加 placement policy 字段和设计文档。
  - 启示:PD 分离、prefill/decode 分组、多角色推理会需要声明式 placement policy。我们的 LLM 部署向导要暴露“角色、切片、拓扑、扩缩策略”，不能只给一个 replicas。
- [LWS PR #910](https://github.com/kubernetes-sigs/lws/pull/910) 与 [#922](https://github.com/kubernetes-sigs/lws/pull/922) — DisaggregatedSet per-role HPA / RoleScaler。
  - 启示:推理服务的 prefill、decode、router、cache 角色扩缩容信号不同。我们应支持按角色配置指标和上下限。
- [LWS PR #919](https://github.com/kubernetes-sigs/lws/pull/919)、[#920](https://github.com/kubernetes-sigs/lws/pull/920)、[#921](https://github.com/kubernetes-sigs/lws/pull/921) — DisaggregatedSet / LeaderWorkerSet topology-aware scheduling 和 CompositePodGroup integration KEP。
  - 启示:LWS 正在与拓扑调度和 gang/PodGroup 语义结合。多节点推理如果要稳定落地，必须把拓扑约束、成组调度、滚动更新放在同一个控制面。
- [JobSet PR #1273](https://github.com/kubernetes-sigs/jobset/pull/1273) 与 [#1274](https://github.com/kubernetes-sigs/jobset/pull/1274) — MutablePod resources / mutable suspended job resources KEP。
  - 启示:训练作业资源动态调整会影响 VPA、Kueue admission、gang scheduling 和 checkpoint。我们应关注它与 in-place resize 的交互。

## KEP 动向

- [NFD v0.19.0](https://github.com/kubernetes-sigs/node-feature-discovery/releases/tag/v0.19.0) 发布。
  - 启示:NFD 仍是异构硬件上报的基础。GPU/NPU 产品需要把 NFD 版本、rule、NodeFeatureGroup 与调度策略作为兼容矩阵维护。
- [NFD PR #2255](https://github.com/kubernetes-sigs/node-feature-discovery/pull/2255) — nfd-worker 增加 FeatureAllowList / FeatureDenyList。
  - 启示:硬件特征暴露需要最小化和可控性，避免把过多节点细节泄露给普通租户。
- [NFD PR #2553](https://github.com/kubernetes-sigs/node-feature-discovery/pull/2553) — NFD network topology updater KEP。
  - 启示:网络拓扑也将进入节点特征发现体系。对 RDMA/RoCE/UB 网络敏感的训练和推理调度，需要提前规划拓扑 label 的标准化。
- [scheduler-plugins PR #973](https://github.com/kubernetes-sigs/scheduler-plugins/pull/973) 与 [#982](https://github.com/kubernetes-sigs/scheduler-plugins/pull/982) — NodeTopologyMatch 支持启动后新建 NRT 对象、NodeResourceTopologyMatch 默认 preemption。
  - 启示:拓扑资源对象的动态更新和抢占将影响 NUMA/GPU 拓扑调度。我们要验证调度器在节点拓扑变化后的 cache 与 preemption 行为。
