# K8s 自身核心周报 2026-04-26

窗口:2026-04-19 → 2026-04-26

## 摘要

- [Kubernetes v1.36.0](https://github.com/kubernetes/kubernetes/releases/tag/v1.36.0) 于 2026-04-22 发布,同时 v1.35.4/v1.34.7/v1.33.11 patch 线继续维护。启示:产品支持矩阵要覆盖 1.33-1.36 的多 minor 客户现实。
- [Karpenter v1.12.0](https://github.com/kubernetes-sigs/karpenter/releases/tag/v1.12.0) 于 2026-04-24 发布,本周修复 do-not-disrupt grace period 测试、already-bound pods timestamp、node labels re-sync,并新增 balanced consolidation RFC。启示:节点自动扩缩容正在向“更少打断业务”精细化演进,对 AI 训练/推理很关键。
- [controller-runtime 本周升级到 Kubernetes v1.36.0 依赖](https://github.com/kubernetes-sigs/controller-runtime/commit/557c3147e26a62272c87db88e3aadda9890931f1)。启示:我们自研 operator 要跟随 controller-runtime/client-go 升级,否则新集群兼容性会先在 watch/admission/cache 层冒出来。

## K8s 核心版本节奏

- [Kubernetes v1.36.0](https://github.com/kubernetes/kubernetes/releases/tag/v1.36.0) 是本周核心版本节点。
  - 启示:AI 平台升级验证要覆盖 control plane、node、CSI、CNI、policy、operator SDK/controller-runtime,不能只跑推理 smoke test。
- [Cluster API v1.13.0](https://github.com/kubernetes-sigs/cluster-api/releases/tag/v1.13.0) 本周发布,并开始测试 Kubernetes v1.36/v1.37: [commit](https://github.com/kubernetes-sigs/cluster-api/commit/2a251ece4f12a20ecddffd1f5a51d1b63bcd097f)。
  - 启示:如果我们后续要管理多集群/节点池生命周期,CAPI 的 K8s support matrix 是重要参考。

## 调度(kube-scheduler)

- [Karpenter v1.12.0](https://github.com/kubernetes-sigs/karpenter/releases/tag/v1.12.0) 增强 disruption 相关能力,本周还有 [Balanced consolidation policy RFC](https://github.com/kubernetes-sigs/karpenter/commit/0de9964bc2ff44edb950925c137ddee972098e0e)。
  - 启示:AI workload 的调度体验不只在 kube-scheduler,还取决于节点生命周期控制。我们应把“可打断/不可打断、宽限期、checkpoint 状态”整合成统一策略。

## 节点 & kubelet

- [Kubernetes main 本周继续更新 etcd SDK、metrics-server、in-place flake 等](https://github.com/kubernetes/kubernetes/commits/master/)。
  - 启示:节点侧升级要特别关注 in-place resize、eviction、image GC、device manager 和 runtime 交互。GPU/NPU 节点升级需要独立 e2e。

## 自动扩缩容(VPA / CA / Karpenter)

- [kubernetes/autoscaler 本周修复 VPA admission validation](https://github.com/kubernetes/autoscaler/commit/16b4450f330d87dc6488ba17cf35eba912a1cfeb),并更新 cluster-autoscaler chart 到 [9.57.0](https://github.com/kubernetes/autoscaler/releases/tag/cluster-autoscaler-chart-9.57.0)。
  - 启示:VPA 与 admission/webhook 稳定性直接影响生产可用性。等 in-place resize 在客户集群成熟后,VPA 才真正能成为 AI 平台默认推荐器。
- [Karpenter v1.12.0](https://github.com/kubernetes-sigs/karpenter/releases/tag/v1.12.0) 是本周 autoscaling 最大信号。
  - 启示:GPU 节点池的 consolidation 不能只看成本,还要结合训练 checkpoint 和推理副本 SLA。

## Runtime(containerd / CRI-O / runc)

- [containerd 本周合入 CVE-2026-34986 相关 PR](https://github.com/containerd/containerd/commit/b6a45a187878d47b48ceb112d80acff14ad37f18),并推进 erofs/dm-verity metadata path: [commit](https://github.com/containerd/containerd/commit/31ca4285805096e8db12a4d2b3b9ddc10f3583ed)。
  - 启示:镜像完整性、dm-verity、runtime CVE 会直接影响私有化 AI 节点合规。应纳入节点基线扫描。
- [CRI-O 本周修复 kill segfault](https://github.com/cri-o/cri-o/commit/ff94ddfaf9f1d28cb36d370509f87ba0ae1250f7),并持续 CNI status poll: [commit](https://github.com/cri-o/cri-o/commit/92586a079db4cf08f65db9a2b0946791a45ef020)。
  - 启示:CRI-O 是 OpenShift 客户的关键 runtime,我们对 OAI/RHOAI 客户环境要保留 CRI-O 专项兼容验证。

## API Machinery & Operator 生态

- [controller-runtime update to k8s.io v1.36.0](https://github.com/kubernetes-sigs/controller-runtime/commit/557c3147e26a62272c87db88e3aadda9890931f1),随后合并依赖更新 PR: [commit](https://github.com/kubernetes-sigs/controller-runtime/commit/c8b4b9d61fbddd8924c1075ec2face3aa7a5f768)。
  - 启示:operator 项目应准备 v1.36 client-go/controller-runtime 升级分支,提前跑 envtest/admission/cache 测试。
- [Kubebuilder 本周升级 golangci-lint 到 v2.11.4](https://github.com/kubernetes-sigs/kubebuilder/commit/f89cae84b701c417b9e5788112223fde9defafa1)。
  - 启示:脚手架工具链升级会带来 lint/生成代码差异,适合在小版本窗口统一处理。

## 网络 & 存储

- [Gateway API 本周添加 session persistence conformance test plan](https://github.com/kubernetes-sigs/gateway-api/commit/894161b5a7d1fb6a365c0688ec89bc03dae63497),并移除 SessionPersistence API 中的 idleTimeout: [commit](https://github.com/kubernetes-sigs/gateway-api/commit/b05a78a26d2c00423999dd07d9cbb249e4e66b84)。
  - 启示:模型网关、Playground、MCP API 都需要稳定会话语义。Gateway API 的 session persistence 要跟踪。
- [Cilium 本周修 X-Forwarded-Proto redirect 保护](https://github.com/cilium/cilium/commit/a0c12204745eceb7fa5ec2be631a7ae0d33771d1),并修 BPF ct buffer 栈分配: [commit](https://github.com/cilium/cilium/commit/524a591bc2f731387a346a7fab3806591db7c47b)。
  - 启示:长连接推理流量、Gateway/L7 policy、Hubble 观测都依赖 CNI patch 质量。
- [external-snapshotter 修 VolumeSnapshotContent deletion](https://github.com/kubernetes-csi/external-snapshotter/commit/e8f68491852cd991d97cf6e235a26db30302571e),[external-provisioner 修 duplicate capacity](https://github.com/kubernetes-csi/external-provisioner/commit/23114109c8596c21c08cf9cacab34e999f6b5fd1)。
  - 启示:模型缓存、checkpoint、训练数据 PVC 都依赖 CSI 稳定性,存储组件也应进入 AI 平台周检。

## 策略 & 安全

- [Kyverno v1.17.2](https://github.com/kyverno/kyverno/releases/tag/v1.17.2) 和 [v1.18.0-rc.1](https://github.com/kyverno/kyverno/releases/tag/v1.18.0-rc.1) 本周出现;main 里有 CLI cross-resource apply: [commit](https://github.com/kyverno/kyverno/commit/126e3be8b44c0d16f17b193c6eec477614376c35)。
  - 启示:Notebook egress、模型部署镜像、API key secret、GPU node selector 都适合用 policy-as-code 管。
- [Gatekeeper 本周修 webhook TLS readiness gate](https://github.com/open-policy-agent/gatekeeper/commit/c344f997c67621cda8796d219f5d2ec9b4cd8775),并增加 structured semantic log 字段: [commit](https://github.com/open-policy-agent/gatekeeper/commit/257267493e380b7350848aa0e267c940e4a37df0)。
  - 启示:策略引擎自己的可观测性会影响生产排障,尤其是 admission latency 和 webhook readiness。

## KEP 动向

- [KEP-5709: well-known pod network readiness gate](https://github.com/kubernetes/enhancements/pull/5995) 本周更新。
  - 启示:Pod 网络就绪门对在线推理很有价值,能减少服务发现早于网络可用导致的冷启动异常。
- [KEP-6012: CompositePodGroup API](https://github.com/kubernetes/enhancements/pull/6017) 新增 initial KEP docs。
  - 启示:与多 Pod 组合调度/多节点 workload 表达相关,值得和 JobSet/LWS/Kueue 一起跟。
