# K8s AI 基础设施周报 2026-04-26

窗口:2026-04-19 → 2026-04-26

## 摘要

- [Kubernetes v1.36.0](https://github.com/kubernetes/kubernetes/releases/tag/v1.36.0) 在 2026-04-22 发布,是本周底座大事件。启示:AI 平台的 Kueue/LWS/KServe/GPU Operator/Device Plugin 兼容矩阵要同步升到 v1.36。
- [KServe v0.18.0-rc1](https://github.com/kserve/kserve/releases/tag/v0.18.0-rc1) 本周发布,LLMInferenceService 继续强化 llm-d v0.6 迁移、WVA/KEDA/HPA autoscaling、LWS、ModelCache namespace scope。启示:KServe 已经把 LLM 专用控制面主线化。
- [NVIDIA k8s-device-plugin v0.19.1](https://github.com/NVIDIA/k8s-device-plugin/releases/tag/v0.19.1) 本周发布,[GPU Operator v26.3.1](https://github.com/NVIDIA/gpu-operator/releases/tag/v26.3.1) 在窗口前一天发布。启示:GPU 节点侧组件仍在快速滚动,平台安装器要把 driver/runtime/device-plugin/operator 版本约束做成显式检查。

## GPU / 加速器管理

- [NVIDIA k8s-device-plugin v0.19.1](https://github.com/NVIDIA/k8s-device-plugin/releases/tag/v0.19.1) 发布,本周 commit 主要是 `go-nvlib` 和 Holodeck 依赖更新: [go-nvlib bump](https://github.com/NVIDIA/k8s-device-plugin/commit/6ecfb69788ac33c50b2b52b3201c80a9bff5c60e)。
  - 启示:device-plugin 本身变更不大,但 NVIDIA 底层库升级会影响设备发现、MIG、健康上报。我们应把 device-plugin 版本列入 GPU 节点诊断信息。
- [NVIDIA GPU Operator v26.3.1](https://github.com/NVIDIA/gpu-operator/releases/tag/v26.3.1) 仍是本周 GPU Operator 的关键版本;本周 main 变更以依赖和 CI 更新为主。
  - 启示:GPU Operator 是企业 AI 集群事实标准。我们如果同时支持 NPU/其他加速器,应抽象出统一 accelerator operator 生命周期:安装、升级、健康、资源上报、节点隔离。

## 调度 & 资源分配

### Kueue

- [Kueue v0.17.1](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.17.1) 仍是近窗口关键 patch。本周 main 出现 [LocalQueue Default Maximum Execution Time KEP](https://github.com/kubernetes-sigs/kueue/pull/10596)、SubtreeQuota 更新/回滚、workload uid lookup skills 等变更: [commits](https://github.com/kubernetes-sigs/kueue/commits/main/)。
  - 启示:Kueue 正在从单集群 batch admission 扩展到队列治理、quota 层级和可诊断性。我们做批调度控制台时要展示 cohort/subtree quota/workload uid 这类排障入口。
- [Kueue v0.17.1 release notes](https://github.com/kubernetes-sigs/kueue/releases/tag/v0.17.1) 包含 MultiKueue、TAS、LWS integration、AdmissionChecks、ElasticJobs 修复。
  - 启示:多集群 + 拓扑感知 + LWS 的组合已经进入生产化修补阶段,适合作为我们多节点推理/训练调度默认底座。

### DRA

- 本周 DRA 方向没有新 release 信号,但 [Kubernetes Dynamic Resource Allocation 文档](https://kubernetes.io/docs/concepts/scheduling-eviction/dynamic-resource-allocation/) 仍是设备资源池化路线的核心参考。
  - 启示:device-plugin 仍是当下主流,DRA 是未来资源声明接口。我们的资源模型应预留 ResourceClaim/ClaimTemplate 对接点,避免把 GPU/NPU 分片绑定到单一厂商插件。

### LWS / JobSet / scheduler-plugins

- [LWS 本周出现 DisaggregatedSet API types 初始提交](https://github.com/kubernetes-sigs/lws/commit/c60da2c49a53f76c64a93f17a3a308c3a3eff711)。
  - 启示:LeaderWorkerSet 社区在探索“分离式集合”抽象,与 LLM prefill/decode 分离、多角色 worker 拓扑高度相关。值得跟 llm-d/KServe 的 PD disaggregation 一起看。
- [scheduler-plugins v0.34.7](https://github.com/kubernetes-sigs/scheduler-plugins/releases/tag/v0.34.7) 发布,本周还完成 Kubernetes v1.35 依赖升级: [commit](https://github.com/kubernetes-sigs/scheduler-plugins/commit/8b585efb64a42037bb2239510a97a0aba666e0e5)。
  - 启示:gang/coscheduling 等插件继续跟随 K8s 版本。若产品仍依赖 scheduler-plugins,必须把它纳入 K8s minor 升级测试。
- [JobSet 本周主要是 Kubernetes 依赖和 Go 版本更新](https://github.com/kubernetes-sigs/jobset/commits/main/)。
  - 启示:JobSet 当前是稳定底座信号,新增价值更多来自 Kueue/LWS/Kubeflow Trainer 的组合。

## KEP 动向

- [KEP-6012: CompositePodGroup API](https://github.com/kubernetes/enhancements/pull/6017) 本周新增 initial KEP docs。
  - 启示:CompositePodGroup 可能成为一组相关 Pod 的原生表达,与多节点训练、gang scheduling、分布式推理拓扑直接相关。
- [KEP-4049: Update for Beta](https://github.com/kubernetes/enhancements/pull/6026) 本周更新到 beta 方向。
  - 启示:需要进一步确认其 SIG 和能力边界,若涉及调度/资源语义,可能影响未来 AI workload API 设计。
